# /setup-browser-cookies：会话管理员

## 一、核心定位

`/setup-browser-cookies` 是 `/browse` 角色的专属辅助认证工具，定位是无头浏览器与会话同步器，核心目标是解决测试需要登录的页面时的认证痛点：不用在无头浏览器里手动填写账号密码、处理多因素认证、滑块验证、CAPTCHA 这些复杂的登录流程，一键把用户本地 Chromium 内核浏览器（Chrome、Edge、Brave 等）里已经登录的 Cookie 加密导入到无头浏览器会话里，导入后访问对应网站直接是登录状态，大大提升需要认证场景下的测试效率，同时避免明文保存账号密码的安全风险。

它是测试需要认证的页面的前置工具，和 `/browse` 配合使用，让无头浏览器也能直接复用本地已经登录的会话，不用重复走登录流程，是提升自动化测试效率的关键辅助工具。

## 二、设计理念

这个技能的设计完全继承了"复用已有状态，避免重复劳动"和"安全第一"的最佳实践，背后的核心逻辑是：

> 测试需要登录的页面时，最麻烦的就是认证环节：要填写账号密码、处理多因素认证、滑块验证、CAPTCHA 这些，自动化很难处理，每次测试都要重新登录也很麻烦，而且在脚本里明文保存账号密码还有安全风险。其实用户本地的浏览器已经登录过这些网站了，直接把已经存在的 Cookie 同步过来是最简单高效的解决方案，不需要处理复杂的登录流程，也不用保存敏感的账号信息，安全又高效。

`/setup-browser-cookies` 有两条核心安全原则：1）Cookie 只在本地解密和传输，绝对不会泄露到外部，也不会明文展示给用户，只会显示域名和数量；2）不会读取和传输无关的 Cookie，用户可以选择只导入需要的域名，安全可控。

## 三、标准工作流程

调用 `/setup-browser-cookies` 后，它会按照标准化的流程完成 Cookie 导入：

1. **前置检查阶段**：首先检查当前 `/browse` 会话是不是已经连接了真实浏览器（CDP 模式），如果已经连接了真实浏览器，就不需要导入 Cookie，直接使用真实浏览器的会话即可；然后检查 `/browse` 工具是否已经构建，没有的话自动完成首次构建
2. **导入模式选择阶段**：如果用户指定了要导入的域名（比如 `/setup-browser-cookies github.com`），就直接导入对应域名的 Cookie，不需要打开选择器；否则打开交互式 Cookie 选择器，自动检测本地安装的所有 Chromium 内核浏览器，在用户默认浏览器打开选择页面
3. **用户选择阶段**：用户在选择器页面可以切换不同的浏览器、搜索域名、选择要导入的域名，不需要手动输入，也看不到明文的 Cookie 值，只能看到域名和 Cookie 数量
4. **权限处理阶段**：macOS 系统会自动请求钥匙串访问权限，Linux 系统自动处理 libsecret 加密的 Cookie，用户只需要点击允许即可，不需要手动操作
5. **导入验证阶段**：用户选择完成后，自动把选中的 Cookie 解密导入到 `/browse` 的无头会话里，然后显示导入结果：导入了哪些域名、每个域名多少个 Cookie
6. **生效阶段**：导入的 Cookie 会立即在当前 `/browse` 会话里生效，后续访问对应域名的页面直接就是登录状态，不需要任何额外操作，会话持久化，重启浏览器也不会丢失。

## 四、核心能力

### 1. 多浏览器多平台支持
- 自动检测本地安装的所有 Chromium 内核浏览器：Chrome、Edge、Brave、Vivaldi、Opera 等
- 跨平台支持：完美适配 macOS、Linux、Windows 系统
- 自动适配不同系统的浏览器 Cookie 加密方式：macOS 钥匙串加密、Linux libsecret 加密、Windows DPAPI 加密，不需要用户手动处理加密解密

### 2. 安全导入机制
- 直接从本地浏览器的加密存储里解密 Cookie，不需要用户输入账号密码，也不需要处理登录流程
- Cookie 全程在本地处理，不会上传到任何服务器，也不会明文展示给用户，只会显示域名和 Cookie 数量
- 用户可以选择只导入需要的域名，不会读取其他无关网站的 Cookie，安全可控
- 不会保存导入的 Cookie 到磁盘，只会保存在浏览器内存里，会话结束自动清除

### 3. 灵活的导入方式
- **交互式选择器模式**：默认模式，打开可视化选择页面，用户可以搜索域名、多选导入，适合不知道具体域名或者需要导入多个域名的场景
- **直接指定域名模式**：用户可以直接指定要导入的域名，比如 `/setup-browser-cookies github.com`，不需要打开选择页面，适合自动化脚本场景
- **多浏览器切换**：选择器页面支持切换不同的本地浏览器，导入不同浏览器里的 Cookie，适合多浏览器测试场景

### 4. 自动生效，无需额外操作
- 导入的 Cookie 会立即在当前 `/browse` 会话里生效，不需要重启浏览器，也不需要额外配置
- 会话持久化，导入的 Cookie 会一直保留到浏览器实例停止，后续所有 `/browse` 操作都可以直接使用登录状态
- 支持多次导入，可以随时导入新的域名的 Cookie，不会覆盖已经导入的 Cookie

### 5. 权限友好处理
- macOS 系统自动弹出钥匙串访问请求，用户只需要点击"始终允许"即可，后续导入不需要再授权
- Linux 系统自动处理 libsecret 的权限请求，不需要用户手动操作
- 不会读取浏览器里的其他敏感数据，只读取 Cookie 信息，权限最小化

## 五、典型使用场景

- 测试需要登录的后台页面、用户中心、支付页面、管理系统等需要认证的场景，不用手动走登录流程
- 避免每次测试都要重新登录，处理多因素认证、验证码、CAPTCHA 这些自动化很难处理的认证环节
- 不想在测试脚本里明文保存账号密码，更安全的认证方式，避免密码泄露风险
- 多账号测试场景，导入不同账号的 Cookie 测试不同权限的用户场景，不用来回切换账号登录
- 测试需要特定登录状态的场景，比如测试只有特定用户才能看到的功能，直接导入对应用户的 Cookie 即可
- 自动化测试脚本里的认证环节，不用写复杂的登录脚本，直接导入 Cookie 即可完成认证

## 七、迁移价值

这个角色的设计思路完全可以脱离 Gstack 复用：

- 从本地浏览器直接导入 Cookie 的思路可以复用在自动化测试工具里，完美解决认证环节的痛点，不用写复杂的登录脚本，也不用保存明文密码，大大提升测试效率和安全性
- 交互式域名选择的用户体验很好，用户不用记住复杂的域名，搜索选择即可，适合非技术用户使用
- 跨平台跨浏览器的 Cookie 解密方案可以参考，不用重复开发不同系统不同浏览器的解密逻辑
- 安全第一的设计原则值得借鉴：Cookie 只在本地处理，不泄露、不明文展示、最小权限读取，保护用户的敏感信息
- 核心逻辑是「复用用户已有的登录状态，避免重复的登录流程，同时保证安全」，适用于所有需要认证的自动化测试场景，大大降低认证环节的复杂度。
