# /freeze：边界管理员

## 一、核心定位

`/freeze` 是 Gstack 编辑范围管控的专属角色，定位是代码修改的边界锁，核心目标是限制当前会话的文件编辑范围到指定目录，所有 Edit 和 Write 操作只能在允许的目录下进行，修改目录外的文件会被直接阻止，防止调试、修复问题、开发某个模块的时候不小心改到其他无关的代码，导致引入意料之外的问题，保持改动范围完全可控，特别是调试、修复小问题、专注单个模块开发的时候，可以避免"顺手改"其他模块代码带来的不可控风险，确保所有改动都在预期范围内。

它是改动范围管控的核心工具，给编辑操作加上硬边界，从根本上避免超出范围的误修改，是保持代码改动可控、减少意外 bug 的重要助手。

## 二、设计理念

这个技能的设计完全继承了"最小改动范围"和"可控性优先"的最佳实践，背后的核心逻辑是：

> 开发者在调试或者修改局部功能的时候，很容易顺手改到其他模块的代码：比如调试支付问题的时候看到用户模块的一个小警告顺手改了，最后提交的时候忘记把这部分改动加到提交说明里，或者这部分改动没有经过测试，导致上线后出问题；或者重构某个模块的时候不小心改到了其他模块依赖的公共方法，引发连锁 bug。这种意料之外的改动是很多线上问题的根源，而且很难排查。给编辑范围加一个硬边界，只能修改指定目录下的文件，其他目录的修改直接被阻止，就可以从根本上避免这类问题，确保所有改动都在预期的范围内。

`/freeze` 有两条核心原则：1）硬阻止而不是警告，超出范围的编辑操作直接被拒绝，从根本上避免误改；2）只限制编辑操作，不影响读取、执行、搜索等其他操作，不会阻碍正常的调试和开发工作。

## 三、标准工作流程

调用 `/freeze` 后，它会按照标准化的流程设置编辑边界：

1. **路径询问阶段**：询问用户要限制的编辑目录路径，用户输入需要的目录（相对路径或者绝对路径都可以）。
2. **路径解析阶段**：把用户输入的路径解析成绝对路径，加上末尾斜杠，避免前缀匹配错误（比如`/src`不会匹配到`/src-old`）。
3. **状态保存阶段**：把解析后的允许编辑目录保存到会话状态文件，会话级存储，重启会话就失效。
4. **钩子挂载阶段**：挂载 PreToolUse 钩子，所有 Edit 和 Write 操作执行前都会触发边界检查。
5. **边界检查阶段**：
   - 每次执行 Edit 或 Write 操作前，自动检查目标文件路径是不是在允许的目录下
   - 如果在允许范围内，正常执行操作
   - 如果超出范围，直接阻止操作，提示用户超出了允许编辑的目录范围
6. **解除限制阶段**：用户可以运行`/unfreeze`解除限制，或者结束会话自动失效，也可以重新运行`/freeze`设置新的允许目录。

## 四、核心能力

### 1. 硬边界限制，从根本上避免误改
- 超出允许目录的编辑操作直接被阻止，不是警告，从根本上避免误改其他目录的文件
- 没有绕过的余地，确保所有改动都在预期范围内
- 比提醒警告更可靠，不会因为用户忽略警告而误改

### 2. 最小干扰，不影响正常开发
- 只限制 Edit 和 Write 两种编辑操作，读取、执行、搜索、替换、终端命令等其他操作完全不受影响
- 不影响正常的调试和开发工作，只是防止编辑其他文件
- 适合调试场景，你可以读所有代码、运行所有命令，只是不能改其他目录的文件

### 3. 智能路径匹配，避免误匹配
- 自动给允许目录加上末尾斜杠，避免前缀匹配错误，比如允许目录是`/src/pay/`的时候，不会匹配到`/src/payment/`目录
- 支持相对路径和绝对路径输入，自动解析成标准绝对路径
- 路径解析准确，不会出现匹配错误的情况

### 4. 灵活可控，随时调整
- 可以随时重新运行`/freeze`设置新的允许编辑目录，调整边界
- 可以运行`/unfreeze`随时解除限制
- 会话级生效，结束会话自动失效，不会对系统造成持久化影响
- 状态保存在本地，不会影响其他会话

### 5. 明确提示，用户清楚知道限制范围
- 设置完成后明确告诉用户允许编辑的目录路径
- 阻止操作时明确提示超出了允许的编辑范围，以及当前允许的目录是什么
- 用户不会困惑为什么操作被阻止，清楚知道当前的限制范围

## 五、典型使用场景

- 调试某个特定模块时启用，防止调试过程中顺手改到其他模块的代码，保持改动范围可控
- 修复单个小 bug 时启用，确保改动只涉及相关的代码文件，避免引入无关的改动
- 开发单个功能模块时启用，专注当前模块，不会不小心改到其他模块的代码
- 多人协作场景下，每个开发者只允许修改自己负责的目录，避免改到别人的代码
- 新手开发者启用，防止不小心改到核心代码或者其他不熟悉的模块
- 做定向性能优化、重构时启用，只允许修改目标目录的代码，避免无关的改动混入
- 排查线上问题时启用，只允许修改问题相关的代码，避免顺手改其他问题导致引入新的 bug

## 六、实际使用示例

> **用户输入**：我要调试支付模块的问题，帮我设置只能修改支付相关的目录。
>
> **/freeze 输出**：
> 请问要限制编辑的目录路径是什么？
>
> （用户回复：src/services/pay）
> 解析路径为绝对路径：/workspace/project/src/services/pay/
> ✅ 编辑边界已设置，现在只能修改 `/workspace/project/src/services/pay/` 目录下的文件，修改其他目录的文件会被阻止。
> 可以随时运行 `/freeze` 重新设置边界，或者运行 `/unfreeze` 解除限制。
>
> （用户尝试修改 src/services/user/user.js）
> ❌ 操作被阻止：文件路径 `/workspace/project/src/services/user/user.js` 超出了允许编辑的目录范围，当前允许编辑的目录是 `/workspace/project/src/services/pay/`。
>
> （用户修改 src/services/pay/callback.js）
> ✅ 操作成功，文件在允许编辑范围内。
>
> （用户运行 /unfreeze）
> ✅ 编辑限制已解除，现在可以修改所有目录的文件。

## 七、迁移价值

这个角色的设计思路完全可以复用在开发工具和团队流程里：

- 编辑范围限制的思路可以集成到 IDE 或者代码编辑器里，支持设置当前工作的目录范围，超出范围的编辑会被阻止或者警告，避免开发者不小心改到其他模块的代码
- 调试、修复 bug 时开启范围限制，可以确保改动范围可控，减少引入意外问题的风险，降低线上事故的概率
- 对于权限管控的场景，可以限制不同角色的用户只能修改自己负责的目录，防止越权修改核心代码或者其他人负责的模块
- 最小改动范围的理念可以推广到整个研发流程，每次提交的改动尽量限制在最小范围，方便代码评审和问题排查
- 核心逻辑是「给编辑操作加硬边界，确保所有改动都在预期范围内，避免意料之外的改动带来的风险」，适用于所有需要保持改动可控的开发场景，是提升代码质量、减少意外 bug 的低成本有效手段。
