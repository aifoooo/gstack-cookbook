# /careful：安全运维工程师

## 一、核心定位

`/careful` 是 Gstack 操作安全防护的专属角色，定位是破坏性命令的智能安全护栏，核心目标是防止误执行危险命令导致的数据丢失、生产事故、代码历史丢失等严重问题，在执行所有 bash 命令前自动检测破坏性操作模式，如果检测到危险操作就会弹出明确的风险警告，让用户确认后才会执行，是操作生产环境、重要数据、共享环境、Git 主分支时的必备安全保障，避免"删库跑路"、"强制推送覆盖主分支"这类低级但后果极其严重的操作失误。

它是操作层面的最后一道安全防线，用几乎可以忽略的成本，避免可能造成巨大损失的误操作，是所有开发者运维操作时的安全助手。

## 二、设计理念

这个技能的设计完全继承了"安全左移"和"最小成本防护"的最佳实践，背后的核心逻辑是：

> 所有开发者都会有手滑的时候，特别是加班疲劳、注意力不集中的时候，很容易执行错误的命令：比如 rm -rf 的时候路径多打了个/、git push -f 的时候忘记选分支、DROP TABLE 的时候表名写错、kubectl delete 的时候选了生产环境的 Namespace，这些失误操作的时间只有几秒钟，但造成的损失可能是几个小时甚至几天的工作，严重的会导致线上事故、数据永久丢失。在执行危险命令前加一层简单的确认，成本只有几秒钟，却可以避免几乎所有这类低级失误导致的损失。

`/careful` 有两条核心原则：1）不阻止用户执行任何操作，只是在执行危险操作前提醒用户确认，给用户纠正失误的机会；2）不影响正常操作，常见的安全删除（比如 node_modules、dist 这类构建目录）不会触发警告，不会给正常开发带来负担。

## 三、标准工作流程

启用 `/careful` 模式后，所有 bash 命令执行前都会自动触发安全检查，流程完全透明不影响正常操作：

1. **钩子挂载阶段**：启用 careful 模式后，自动挂载 PreToolUse 钩子，所有 Bash 命令执行前都会先经过安全检查脚本，会话级生效，结束会话就自动失效，不需要手动关闭。
2. **命令检查阶段**：执行 bash 命令前，检查脚本会自动检测命令内容：
   - 首先匹配破坏性命令模式，判断是不是危险操作
   - 如果匹配危险模式，再检查是不是安全例外的操作（比如删除 node_modules、dist 这类缓存构建目录）
   - 如果是真正的危险操作，拦截命令执行，弹出警告
3. **风险提示阶段**：检测到危险操作后，弹出明确的警告，说明具体的风险类型（递归删除数据丢失、改写 Git 历史、删除生产资源等），让用户选择继续执行还是取消。
4. **执行决策阶段**：用户确认继续的话正常执行命令，否则取消命令执行，不会造成任何影响。
5. **安全操作直通阶段**：如果命令是安全的，或者是安全例外的操作，直接执行，不会有任何提示，不影响正常操作效率。

## 四、核心能力

### 1. 全场景危险操作覆盖
覆盖所有常见的高风险操作类型：
- **文件删除类**：`rm -rf`、`rm -r`、递归删除操作，风险是数据丢失
- **数据库操作类**：`DROP TABLE`、`DROP DATABASE`、`TRUNCATE`，风险是数据永久丢失
- **Git 操作类**：`git push --force`、`git reset --hard`、`git checkout .`，风险是代码历史丢失、未提交工作丢失
- **K8s 运维类**：`kubectl delete`类操作，风险是生产环境服务中断
- **Docker 操作类**：`docker rm -f`、`docker system prune -a`，风险是容器、镜像丢失
- 持续更新常见危险操作模式，覆盖更多场景

### 2. 智能安全例外机制
不会对正常开发操作造成干扰：
- 删除常见的构建、缓存目录不会触发警告：`node_modules`、`.next`、`dist`、`__pycache__`、`.cache`、`build`、`.turbo`、`coverage`
- 这些目录都是可以重新生成的，删除不会造成永久损失，直接放行不警告
- 例外列表持续更新，覆盖更多常见的安全删除场景

### 3. 非侵入式设计，无额外负担
- 启用后不需要修改任何操作习惯，所有命令正常执行，只有危险操作前会提示
- 会话级生效，不想用了结束会话或者开新会话就自动失效，不需要额外关闭操作
- 不修改任何系统配置，不会对系统造成任何持久化影响
- 提示简洁，只说明风险类型，不会有多余的信息打扰用户

### 4. 明确的风险提示
用户可以清楚知道操作的后果：
- 警告会明确说明检测到的操作类型和对应的风险：比如"检测到递归删除操作，风险：数据丢失"
- 不会用模糊的提示让用户困惑，明确告知可能的后果
- 让用户在确认前清楚知道自己在做什么，避免误确认

### 5. 灵活可控，用户有最终决策权
- 永远不会阻止用户执行任何操作，只是提醒确认，用户可以选择继续或者取消
- 对于确认要执行的危险操作，不会做额外的阻拦
- 完全尊重用户的决策，只是加一层确认避免失误

## 五、典型使用场景

- 操作生产环境、线上数据库时启用，防止误操作导致线上事故
- 操作重要数据、不可恢复的文件时启用，避免数据丢失
- 多人共享的开发环境、服务器上启用，防止互相误操作
- 新手开发者、实习生操作服务器时启用，避免因为不熟悉命令导致失误
- 操作 Git 主分支、重要仓库时启用，防止强制推送、硬重置导致代码历史丢失
- 加班、疲劳工作时启用，防止注意力不集中导致手滑失误
- 执行删除、修改类的批量操作前启用，确认操作范围正确

## 六、实际使用示例

> **用户输入**：开启 careful 模式，我要操作生产服务器。
>
> **/careful 输出**：
> ✅ 安全模式已启用，所有 bash 命令执行前会检查破坏性操作，检测到危险命令会提示确认。
> 当前保护的危险操作：递归删除、数据库删表/清空、Git 强制推送/硬重置、K8s 删除操作、Docker 删除操作。
> 安全例外目录：node_modules、dist、build、cache 等构建缓存目录。
>
> （用户执行命令：rm -rf /data/backup）
> ⚠️ 安全警告：检测到递归删除操作，风险：数据丢失，是否确认继续执行？（Y/N）
> 用户回复 Y → 执行命令。
>
> （用户执行命令：rm -rf node_modules）
> 识别为安全例外操作，直接执行，无警告。
>
> （用户执行命令：git push -f origin main）
> ⚠️ 安全警告：检测到 Git 强制推送操作，风险：改写远程分支历史，是否确认继续执行？（Y/N）
> 用户回复 N → 命令取消执行，无任何影响。
>
> （用户执行命令：kubectl delete pod -n prod --all）
> ⚠️ 安全警告：检测到 K8s 删除操作，风险：生产环境服务中断，是否确认继续执行？（Y/N）

## 七、迁移价值

这个角色的设计思路完全可以复用在团队的运维和开发流程里：

- 前置安全检查的思路可以复用在所有操作流程里，给所有危险操作加一层确认环节，用极小的成本避免极大的损失，特别是生产环境、重要数据的操作，必须要有确认环节
- 智能例外机制可以平衡安全和效率，不会因为防护措施影响正常工作，避免安全措施变成阻碍工作的负担
- 会话级生效的设计灵活可控，需要的时候启用，不需要的时候自动失效，不会造成持久化的影响
- 明确的风险提示可以避免用户不知道操作后果的情况，让用户做决策前清楚知道可能的后果
- 核心逻辑是「在不影响正常操作的前提下，给危险操作加最后一道确认防线，避免低级失误导致的严重损失」，适用于所有需要操作重要资源的场景，是成本极低但效果极好的安全防护手段。
