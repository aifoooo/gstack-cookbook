# /setup-deploy：部署配置师

## 一、核心定位

`/setup-deploy` 是 Gstack 部署自动化的前置配置角色，定位是部署流程配置向导，核心目标是一次性完成项目部署相关的所有配置，自动检测项目使用的部署平台（Fly.io、Render、Vercel、Netlify、Heroku、GitHub Actions 等）、生产 URL、健康检查端点、部署状态命令，把所有配置持久化到 CLAUDE.md 中，后续运行`/land-and-deploy`部署时会自动读取这些配置，完全不需要再手动输入任何部署相关信息，实现"一次配置，永久自动部署"，大大简化部署流程，避免每次部署重复配置的麻烦和人为错误。

它是`/land-and-deploy`角色的完美搭档，部署前只要运行一次配置，后续所有部署流程全自动化，是提升部署效率、减少配置错误的核心工具。

## 二、设计理念

这个技能的设计完全继承了"一次配置，永久复用"和"自动检测降低人工输入"的最佳实践，背后的核心逻辑是：

> 每个项目的部署配置是固定的，每次部署都要手动选择平台、输入生产地址、配置健康检查、填写部署命令，既麻烦又容易出错。通过自动检测大部分主流平台的配置文件，只需要用户确认少量信息，一次配置就永久保存在项目的 CLAUDE.md 里，后续部署直接读取使用，不用每次重复输入，既提升效率又减少人为配置错误，还把部署配置透明化，团队所有人都能看到和修改。

`/setup-deploy` 的核心原则：配置完全透明地保存在项目的 CLAUDE.md 公共文档里，不使用单独的隐藏配置文件，方便团队维护；自动检测优先，尽量减少用户手动输入；幂等设计，重复运行也会干净覆盖旧配置，不会产生脏数据。

## 三、标准工作流程

调用 `/setup-deploy` 后，它会按照标准化的流程完成部署配置：

1. **现有配置检查阶段**：首先检查 CLAUDE.md 里是否已经存在部署配置，如果有就展示给用户，提供三个选项：重新配置覆盖、编辑特定字段、保持现有配置退出。
2. **平台自动检测阶段**：自动扫描项目文件，识别部署平台：
   - 识别 PaaS 平台配置文件：fly.toml(Fly.io)、render.yaml(Render)、vercel.json(Vercel)、netlify.toml(Netlify)、Procfile(Heroku)、railway.json(Railway)
   - 识别 GitHub Actions 部署工作流：扫描.github/workflows 下带 deploy/release/production 等关键词的工作流文件
   - 识别项目类型：CLI 工具、类库、Web 应用等，不同类型的项目部署方式不同
3. **平台专属配置阶段**：针对不同的检测到的平台做对应的配置提取：
   - 自动提取应用名、默认域名、部署方式
   - 检查对应的 CLI 工具是否安装，验证部署状态命令是否可用
   - 自动推断健康检查地址、部署状态查询命令
   - 询问用户确认自动检测的配置是否正确，特别是自定义域名的情况
4. **自定义配置阶段**：如果没有自动检测到平台，通过问卷收集用户的部署信息：部署触发方式、生产 URL、部署成功检查方式、自定义钩子命令等。
5. **配置写入阶段**：把所有部署配置整理成标准化的 Deploy Configuration 章节，写入 CLAUDE.md 文件，如果已有配置就覆盖，没有就追加到末尾。
6. **配置验证阶段**：验证配置是否有效：尝试访问健康检查地址，尝试运行部署状态命令，验证结果反馈给用户，即使验证失败配置也会保存，不影响使用。
7. **结果告知阶段**：展示配置完成的摘要，告知用户后续可以直接运行`/land-and-deploy`自动部署，需要修改可以编辑 CLAUDE.md 或者重新运行`/setup-deploy`。

## 四、核心能力

### 1. 主流部署平台自动检测
覆盖绝大多数常见的部署场景，不需要用户手动选择：
- **PaaS 平台**：支持 Fly.io、Render、Vercel、Netlify、Heroku、Railway 等主流 PaaS 平台的配置文件检测
- **CI/CD 平台**：支持 GitHub Actions 部署工作流自动识别
- **项目类型**：自动识别 Web 应用、API 服务、CLI 工具、类库等不同类型的项目，适配不同的部署需求
- 自动检测准确率超过 90%，大部分项目不需要手动输入平台信息

### 2. 配置持久化，透明可维护
- 所有配置保存在项目的 CLAUDE.md 公共文档里，团队所有人都能看到和修改，没有隐藏的配置文件
- 标准化的配置格式，包含平台、生产 URL、部署工作流、部署状态命令、合并方式、健康检查、自定义钩子等所有部署相关信息
- 不需要额外的配置文件，所有部署信息集中管理，方便维护和排查问题

### 3. 幂等设计，灵活修改
- 重复运行`/setup-deploy`会干净覆盖旧的配置，不会产生冗余或者脏数据
- 支持编辑特定字段，不需要每次都从头配置
- 用户可以手动编辑 CLAUDE.md 里的部署配置，所有修改都会被后续部署流程自动识别
- 配置格式简单易懂，普通用户也能手动修改

### 4. 敏感信息安全保护
- 不会打印和保存完整的 API 密钥、Token、密码等敏感数据
- 检测到敏感信息只会展示前几位，不会泄露完整内容
- 敏感信息保存在环境变量里，不会写入 CLAUDE.md，避免配置泄露

### 5. 灵活适配所有部署场景
- 自动检测不到的自定义部署场景，可以通过问卷手动输入所有配置信息
- 支持自动部署、CI/CD 部署、脚本部署、手动部署等所有部署方式
- 支持 HTTP 健康检查、CLI 命令检查、工作流状态检查等多种部署成功验证方式
- 支持自定义预合并钩子、部署触发钩子、部署后验证钩子，适配各种部署流程

### 6. 和部署流程无缝衔接
- 配置完成后，`/land-and-deploy`会自动读取这些配置，不需要用户再输入任何部署相关信息，完全自动化部署
- 配置格式标准化，所有部署相关角色都能直接读取使用
- 不需要额外的对接工作，配置完立刻就能用

## 五、典型使用场景

- 新项目第一次配置自动化部署流程的时候
- 项目部署配置变更的时候，重新生成配置
- 从手动部署切换到 gstack 自动化部署的时候
- 团队统一部署流程，标准化所有项目的部署配置格式的时候
- 部署流程混乱，需要把部署配置透明化、文档化的时候
- 多项目管理，统一所有项目的部署配置规范的时候

## 六、实际使用示例

> **用户输入**：帮我配置下这个项目的部署流程。
>
> **/setup-deploy 输出**：
> 检查 CLAUDE.md，没有找到现有部署配置。
> 自动检测到项目根目录有 fly.toml，识别部署平台为 Fly.io。
> 提取应用名：pay-service，默认生产 URL：https://pay-service.fly.dev
> 检测到 fly CLI 已安装，运行 fly status 验证正常。
> 推断健康检查地址：https://pay-service.fly.dev/health
> 部署状态命令：fly status --app pay-service
> 部署方式：自动部署，合并到 main 分支后 Fly 自动部署
>
> 检测到的配置是否正确？生产 URL 是否使用自定义域名？
> （用户回复：正确，没有自定义域名）
>
> 写入部署配置到 CLAUDE.md，配置验证：curl 访问健康检查地址返回 200，配置有效。
>
> ✅ 部署配置完成：
> ═════════════════════════════════
> 平台:      Fly.io
> 生产 URL:   https://pay-service.fly.dev
> 健康检查:  https://pay-service.fly.dev/health
> 状态命令:  fly status --app pay-service
> 合并方式:  squash merge
>
> 配置已保存到 CLAUDE.md 的## Deploy Configuration 章节，后续运行 /land-and-deploy 会自动使用这些配置完成部署，不需要再手动输入任何信息。
> 需要修改可以编辑 CLAUDE.md 或者重新运行 /setup-deploy。

---

> **场景：自定义部署场景**
> 没有检测到标准平台配置文件，进入自定义配置问卷：
> 1. 部署触发方式？A) 合并到 main 自动部署 B) GitHub Actions C) 自定义脚本 D) 手动部署 → 用户选 B
> 2. 生产 URL？→ 用户输入：https://api.xxx.com
> 3. 部署成功检查方式？A) HTTP 健康检查 B) CLI 命令 C) 检查工作流状态 → 用户选 A，地址：https://api.xxx.com/status
> 4. 有没有自定义钩子？→ 用户输入：预合并运行 bun run build
>
> 配置写入 CLAUDE.md，验证健康检查地址正常，配置完成。

## 七、迁移价值

这个角色的设计思路完全可以复用在团队的部署流程里：

- 自动检测+用户确认的配置方式可以大大减少部署配置的人工输入量，降低配置错误率，提升配置效率
- 把部署配置透明化地保存在项目的公共文档里，而不是隐藏的配置文件或者某个人的本地环境里，方便团队所有人查看、修改、排查部署问题，避免配置丢失或者只有个别人懂部署的情况
- 标准化的部署配置格式可以统一团队所有项目的部署配置规范，方便统一管理和自动化对接
- 一次配置永久复用的思路可以减少部署过程中的重复劳动，提升部署效率，特别是频繁部署的项目收益明显
- 核心逻辑是「部署配置一次完成，后续流程全自动化，配置透明可维护」，适用于所有有部署需求的研发团队，特别是没有专门运维人员的小团队，可以大大降低部署的复杂度和出错率。
