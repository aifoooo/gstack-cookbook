# /qa-only：测试专员

## 一、核心定位

`/qa-only` 是 Gstack 中立测试报告的专属角色，完全复刻了独立测试工程师的纯测试逻辑，它的定位就是第三方中立质量审计官，核心目标是只做测试和问题记录，**绝对不修改任何代码**，输出客观、完整、带证据的测试报告，专门用于测试和修复权责分离的场景，避免自动修复带来的不可控风险。

和 `/qa`（测试+修复一体化）不同，它是纯报告模式，全程不触碰源代码，只客观记录发现的问题和复现证据，是需要独立测试报告、合规审计、第三方质量检测场景的首选工具。

## 二、设计理念

这个技能的设计完全继承了行业"测试与修复权责分离"的合规最佳实践，背后的核心逻辑是：

> 在很多正式场景下，测试和修复必须分离：上线前的合规测试需要独立第三方出具报告，不能同时既当运动员又当裁判员；有些团队的流程要求测试只负责找bug，修复交给对应的开发人员，避免自动修复带来的逻辑风险和权责不清。纯报告模式的测试可以保证完全中立，客观记录所有问题，不会因为自动修改代码引入新的风险。

`/qa-only` 有一条不可动摇的铁律：**只发现和记录问题，绝对不修改任何代码，也不主动提供修复建议**，保证测试结果的中立性和客观性。

## 三、标准工作流程

调用 `/qa-only` 后，它会按照标准化的流程开展中立测试，不会跳过任何关键环节：

1. **前置准备阶段**：确认测试目标URL、测试层级（快速/标准/全面）、测试范围、认证信息，如果是在功能分支上会自动进入diff感知模式，只测试代码改动影响到的页面和功能
2. **环境准备**：自动检测本地运行的服务，导入用户cookie或者自动登录，准备测试所需的权限
3. **应用地图探索**：先访问首页，爬取所有可访问的页面和导航路径，形成测试地图，优先测试核心功能路径
4. **逐页测试阶段**：按照测试地图逐个页面测试，每个页面执行完整的检查清单：视觉扫描、交互元素测试、表单测试、导航测试、状态测试（空/加载/错误）、控制台错误检查、响应式测试
5. **问题实时记录**：发现问题立刻记录证据，包括截图、复现步骤、错误信息，每个问题都有可复现的完整证据，全程不做任何代码修改
6. **健康评分计算**：按照统一的8大类加权评分体系计算健康总分，所有打分都有明确的问题支撑
7. **测试报告输出**：输出完整的中立测试报告，包含发现的问题、严重等级、复现步骤、所有证据截图、健康评分、后续建议，全程不提供修复方案，也不修改任何代码
8. **基线保存**：自动保存测试基线，方便后续做回归测试对比

## 四、核心能力

### 1. 完全中立的测试能力
全程不触碰任何源代码，不提供修复建议，只客观记录问题：
- 没有自动修复逻辑，不会修改任何文件
- 报告里只描述问题、严重等级、复现步骤，不给出修复方案
- 所有问题都有完整的截图证据，完全客观中立，适合合规审计场景

### 2. 多模式灵活测试能力
和 `/qa` 支持完全相同的测试模式：
- **diff感知模式**：功能分支自动识别代码改动，只测试影响到的页面和功能
- **全量模式**：全站完整测试，覆盖所有可访问页面
- **快速模式**：30秒冒烟测试，只测首页+核心5个页面
- **回归模式**：和之前的测试基线对比，看哪些问题修好了，哪些是新出现的问题

### 3. 全维度问题检测能力
和 `/qa` 使用完全相同的8大类测试维度和加权评分体系，保证测试标准一致：
- 控制台错误（权重15%）：JS错误、接口报错等
- 链接有效性（权重10%）：死链、404错误等
- 视觉问题（权重10%）：布局错乱、样式错误、显示异常等
- 功能问题（权重20%）：按钮失效、表单提交失败、流程走不通等
- 体验问题（权重15%）：交互卡顿、反馈不及时、操作反人类等
- 性能问题（权重10%）：加载慢、卡顿、资源过大等
- 内容问题（权重5%）：错别字、文案错误、空状态提示不合理等
- 可访问性问题（权重15%）：对比度不够、键盘导航不支持、屏幕阅读器适配问题等

### 4. 多框架专项适配能力
和 `/qa` 支持完全相同的框架专项检查，保证测试深度一致：
- **Next.js**：检查水合错误、客户端路由问题、数据请求错误
- **Rails**：检查N+1查询、CSRF token问题、Turbo交互问题
- **WordPress**：检查插件冲突、REST API问题、混合内容警告
- **SPA应用**：检查客户端路由、状态管理、浏览器前进后退问题

### 5. 标准化报告输出能力
输出标准化的测试报告，完全符合团队协作和合规要求：
- 所有问题都有对应的截图证据和复现步骤
- 统一的健康评分体系，测试结果可以跨版本、跨团队对比
- 报告结构规范，方便开发团队按照报告修复问题
- 自动保存测试基线，支持后续回归测试对比

### 6. 测试基础设施检测能力
自动检测项目的测试基础设施，如果发现没有测试框架，会在报告里提示可以用 `/qa` 来自动化搭建测试框架和生成回归测试用例。

## 五、典型使用场景

- 上线前的独立质量审计，需要中立第三方测试报告
- 第三方合规检测场景，不允许测试人员修改代码
- 团队流程要求测试只负责找bug，修复交给对应开发人员
- 需要留存正式测试报告作为上线依据的合规场景
- 不想让AI自动修改代码，只想知道有什么问题的时候
- 大版本上线前的最终确认测试，需要正式的测试报告存档

## 七、迁移价值

这个角色的设计思路完全可以脱离 Gstack 复用：

- "测试与修复权责分离"的模式可以复用在需要合规审计的团队测试流程里，独立测试角色不接触代码修改，保证测试的中立性和客观性，满足合规要求
- 标准化的中立测试报告模板可以直接作为团队的测试报告规范，统一不同测试人员的输出格式，方便开发团队修复问题
- 完全中立的测试定位可以复用在第三方质量检测场景，测试方只负责客观记录问题，不参与修复，保证测试结果的可信度
- 核心逻辑是「测试角色保持完全中立，只客观记录问题，不修改代码，不提供修复建议」，适用于所有需要独立测试报告的研发场景
