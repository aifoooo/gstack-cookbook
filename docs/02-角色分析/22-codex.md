# /codex：技术评审专家

## 一、核心定位

`/codex` 是 Gstack 多AI交叉评审的专属角色，是OpenAI Codex的CLI封装工具，定位是独立第二意见专家，核心目标是提供来自不同AI系统的技术评审，作为Claude评审的补充，以对抗性思维找出代码、计划、方案中的漏洞、边缘场景、安全问题、性能隐患，避免单个AI模型的偏见和盲区，提升评审的准确性和全面性。它的人设是"200IQ耿直技术宅开发者"，直接、简洁、技术精准，不讲情面，只关注问题，专门挑战假设，找出常规评审容易遗漏的问题，是高风险代码、重要技术方案评审的必备交叉验证工具。

它是双模型评审机制的核心，重要代码和方案同时经过Claude和Codex两个不同AI模型的评审，大大降低问题遗漏的概率，是代码质量和方案可行性的双重保障。

## 二、设计理念

这个技能的设计完全继承了"交叉验证减少盲区"和"第二意见降低风险"的最佳实践，背后的核心逻辑是：

> 单个AI模型和单个评审者一样，总会有自己的偏见和盲区，特别是如果评审模型和生成代码/方案的模型是同一个体系的，更容易有相同的思维盲区，很多问题发现不了。引入训练背景、设计目标完全不同的AI模型做交叉评审，可以大大降低问题遗漏的概率。Codex的定位就是专门挑问题的"坏人"，直接、不讲情面，只关注技术问题，不会说场面话，能发现很多常规评审忽略的边缘场景、安全漏洞、性能问题。

`/codex` 有一条核心原则：所有输出完全忠实于Codex的原始回复，不做总结、不修改、不润色，直接展示给用户，避免中间转述导致的信息失真，确保用户看到最原始的评审结果。

## 三、标准工作流程

调用 `/codex` 后，它会按照标准化的流程提供第二意见：

1. **前置检查阶段**：检查Codex CLI是否已经安装，如果没有安装提示用户安装命令；检查当前Git仓库，确认要评审的diff范围。
2. **模式识别阶段**：根据用户的输入自动识别运行模式：
   - **评审模式**：用户输入`/codex review`，针对当前分支和基准分支的diff做代码评审，标记问题等级，P1问题直接判定不通过
   - **挑战模式**：用户输入`/codex challenge`，对抗性评审，专门找代码的漏洞、边缘场景、安全问题、失败模式
   - **咨询模式**：用户输入其他内容，会话式咨询，支持上下文延续，用户可以问任何技术问题
3. **边界防护阶段**：所有发给Codex的提示都会加上文件系统边界提示，避免Codex读取Gstack自身的skill文件分心，专注于用户的代码和方案。
4. **模式执行阶段**：
   - **评审模式**：运行Codex代码评审，输出原始评审结果，检测P1问题自动判定通过/不通过，自动和之前Claude的`/review`结果做交叉对比，找出共识和分歧点，保存评审日志供后续流程使用
   - **挑战模式**：用对抗性提示运行Codex，尝试找出所有可能的问题，输出原始结果
   - **咨询模式**：检查是否有活跃会话，用户可以选择继续之前的会话或者开新会话，运行Codex咨询，保存会话ID供后续多轮对话使用
5. **结果展示阶段**：所有输出原封不动展示给用户，不做任何修改、总结、润色，确保信息不失真，补充消耗的Token和预估成本，如果和Claude的评审有分歧明确标注出来。

## 四、核心能力

### 1. 三种模式覆盖所有评审场景
- **评审模式**：针对代码diff做标准代码评审，标记问题等级（P1/P2），有P1问题直接判定FAIL，作为代码合并的门禁
- **挑战模式**：对抗性评审，站在攻击者和混沌工程师的视角，找代码的边缘场景、竞态条件、安全漏洞、资源泄露、失败模式、静默数据损坏路径
- **咨询模式**：会话式问答，可以问任何技术问题，支持多轮对话，上下文自动延续，不用每次重新提供背景信息

### 2. 独立第二意见，交叉评审减少盲区
- 完全独立于Claude的AI模型，不同的训练背景和设计目标，评审视角完全不同
- 可以发现很多Claude忽略的问题，特别是技术细节、边缘场景、安全漏洞类问题
- 自动和Claude的评审结果做交叉对比，找出两个模型的共识和分歧点，方便用户决策
- 双模型评审大大降低问题遗漏的概率，重要代码和方案的评审更可靠

### 3. 对抗性思维，专门找问题
- 挑战模式下Codex会专门找问题，不会有任何表扬和场面话，所有输出都是问题
- 会考虑极端场景、异常输入、高并发情况、恶意攻击等常规评审容易忽略的情况
- 特别适合安全相关代码、核心业务模块、高并发组件的评审，提前发现潜在风险

### 4. 会话延续能力，上下文不丢失
- 咨询模式支持多轮对话，自动保存会话ID，后续对话不用重新提供上下文
- 可以基于之前的回答继续追问，不用每次重复背景信息
- 适合复杂技术问题的逐步深入探讨

### 5. 自动评审门槛，可作为CI门禁
- 评审模式会自动判定通过/不通过，如果有P1级别的问题直接标记FAIL
- 评审结果自动保存到评审日志，可以被`/ship`流程识别，没有通过Codex评审的代码不能合并
- 可以集成到CI/CD流程，作为高风险模块合并的强制门禁

### 6. 原始输出展示，信息零失真
- 所有Codex的输出原封不动展示给用户，不做任何修改、总结、润色
- 避免中间转述导致的信息丢失或者偏差，用户看到的就是Codex的原始回复
- Codex的思考过程也会完整展示，用户可以看到它的推理路径，更方便理解问题的由来

### 7. 灵活的推理力度配置
- 支持不同的推理力度配置：默认评审/挑战模式用high，咨询模式用medium
- 支持`--xhigh`参数开启最高推理力度，适合特别重要的代码和方案，推理更彻底，当然耗时和成本也更高
- 自动开启缓存版网页搜索，Codex可以查找最新的文档和API信息，评审更准确

## 五、典型使用场景

- 重要功能、核心业务模块、安全相关代码PR合并前的交叉评审，确保没有严重问题
- 安全敏感模块（支付、认证、数据处理）的对抗性评审，找出潜在的安全漏洞
- 复杂技术方案、架构设计的第二意见，验证方案的可行性和合理性
- 技术问题咨询，获得不同视角的解答和建议
- 高风险版本上线前的多轮评审，确保上线后不会出现严重问题
- 和Claude的评审结果有分歧时，用Codex做第三方评审，辅助决策
- 没有多余人力做交叉评审的小团队，用AI做交叉评审提升代码质量

>
> **/codex 输出**：
> CODEX SAYS (adversarial challenge):
> ════════════════════════════════════════════════════════════

> 1. 并发场景下，同一个订单同时发起多次支付请求，会生成多个支付订单，导致重复扣款。
> 2. 回调接口的签名校验没有做时间窗口校验，攻击者可以重放历史回调请求，重复触发支付成功逻辑。

> 4. 退款接口没有校验退款金额不能大于原支付金额，攻击者可以申请超过原支付金额的退款。

## 七、迁移价值

这个角色的设计思路完全可以复用在团队的研发流程里：

- 多AI交叉评审的思路可以复用在团队的评审流程里，重要代码和方案用不同的AI模型做评审，减少单个模型或者单个评审者的偏见和遗漏，提升评审质量
- 对抗性评审的方式可以作为安全评审的标准流程，专门安排角色站在攻击者的视角找漏洞和边缘场景，提前发现潜在的安全风险
- 第二意见机制可以降低决策风险，重要的技术方案和架构设计都找不同视角的专家做评审，避免思维盲区导致的决策失误
- 原始输出展示的原则可以避免中间转述导致的信息失真，评审过程透明可追溯，用户可以看到最原始的评审意见
- 交叉对比的方式可以明确不同评审者的共识和分歧点，方便团队讨论和决策
- 核心逻辑是「引入不同视角的独立评审，减少盲区，降低风险」，适用于所有需要高质量评审的研发场景，特别是安全和核心业务相关的模块。
