# /browse：测试工程师

## 一、核心定位

`/browse` 是 Gstack 所有浏览器相关操作的底层核心角色，完全封装了高性能无头 Chromium 引擎，它的定位是自动化测试的"瑞士军刀"，核心目标是让开发者不用学习复杂的 Puppeteer/Playwright 框架，不用写复杂的测试脚本，一条简单命令就能完成浏览器导航、交互、验证、截图所有操作，速度极快（启动约 3 秒，后续每个命令仅 100ms），状态持久化（Cookie、登录状态、标签页永久保留），覆盖 90% 的常见测试场景，是所有测试类角色（/qa、/canary、/design-review、/benchmark）的底层能力提供者，也是开发者快速验证功能、复现 Bug、收集证据的首选工具。

它把传统复杂的无头浏览器操作封装成简单的自然命令，让没有测试经验的开发者也能快速完成自动化测试、功能验证、Bug 复现，大大降低自动化测试的门槛。

## 二、设计理念

这个技能的设计完全继承了"工具要简单好用，解决 90% 的常见场景，剩下 10% 开放出口"的最佳实践，背后的核心逻辑是：

> 传统的无头浏览器工具（Puppeteer、Playwright）功能强大，但配置复杂、API 难记、运行速度慢，小团队很难用起来，大部分开发者还是手动打开浏览器测试。这个工具的设计目标就是做"命令行里的浏览器"，简单、快速、好用，内置所有常用的测试操作，不用写复杂的脚本，一条命令就能完成导航、点击、填写表单、截图这些常用操作，速度快到感觉不到延迟，90% 的场景直接用命令就能搞定，遇到复杂场景（比如 CAPTCHA、MFA）可以一键交给用户手动操作，完成后再自动拿回控制权，兼顾简单性和灵活性。

`/browse` 有两条核心原则：1）速度优先，所有操作必须足够快，不能让用户等；2）90% 的场景开箱即用，剩下 10% 开放出口，不追求 100% 覆盖所有复杂场景，遇到 AI 搞不定的可以交给用户。

## 三、标准工作流程

调用 `/browse` 相关操作后，它会按照标准化的流程执行：

1. **初始化阶段**：检查无头浏览器是否已经构建和启动，没有的话自动完成首次构建（约 10 秒），启动浏览器实例，状态持久化，后续所有操作都复用这个实例，不用每次重新启动
2. **基础操作阶段**：支持所有常见的浏览器操作：
   - 导航：跳转 URL、前进、后退、刷新、获取当前 URL
   - 交互：点击元素、填写表单、选择下拉选项、上传文件、处理对话框、按键、滚动、hover、设置 Cookie、设置请求头、设置 User Agent、设置视口大小
   - 检查：检查元素状态（可见/可用/选中/可编辑/聚焦）、获取控制台错误、获取网络请求、获取性能指标、获取 Cookie、获取本地存储、获取页面内容/HTML/链接/表单
3. **可视化阶段**：支持各种可视化操作：
   - 普通截图：整个页面或者指定元素截图
   - 标注快照：自动识别页面上的交互元素，编号标注，生成带参考号的截图和元素树
   - 响应式测试：自动生成移动端、平板、桌面端三个尺寸的截图
   - 页面对比：对比两个 URL 的页面内容差异
   - PDF 导出：把页面导出为 PDF
4. **高级功能阶段**：
   - 多标签页管理：新建标签、切换标签、关闭标签
   - iframe 切换：切换到 iframe 上下文操作
   - 用户接管：遇到 CAPTCHA、MFA、复杂 OAuth 这些 AI 搞不定的场景，自动打开可见浏览器交给用户操作，用户完成后自动恢复控制权继续执行
   - 状态保存加载：保存浏览器状态（Cookie、URL、登录状态），下次直接恢复不用重新登录
5. **结束阶段**：可以选择停止浏览器实例，或者保持状态供后续操作使用，默认保持状态，提升后续操作速度。

## 四、核心能力

### 1. 极快的运行速度
- 首次启动仅需 3 秒，后续每个操作命令仅需 100ms，远快于传统的 Puppeteer/Playwright
- 浏览器状态持久化，Cookie、登录状态、标签页都保留，不用每次重新启动、重新登录
- 所有命令都是原生实现，没有多余的抽象层，速度接近手动操作浏览器

### 2. 零代码交互能力
不用写复杂的 CSS 选择器和脚本：
- 运行`snapshot -i`自动识别页面上所有交互元素（按钮、链接、输入框），自动编号为@e1、@e2、@e3...
- 直接用编号引用元素操作，比如`click @e3`点击第三个按钮，`fill @e4 "user@test.com"`填写第四个输入框
- 自动识别带 onclick、cursor:pointer 的可点击元素，支持`-C`参数生成@c 编号引用
- 不用记复杂的选择器语法，看快照里的编号就能操作

### 3. 全覆盖的浏览器操作能力
覆盖 90% 的常见测试和验证场景：
- **导航类**：跳转、前进、后退、刷新、自定义请求头、User Agent、Cookie 设置
- **交互类**：点击、输入、选择下拉、上传文件、处理弹窗确认/提示、按键、滚动、hover、等待元素加载/网络空闲
- **检查类**：检查元素是否可见/可用/选中/可编辑/聚焦、获取控制台错误、获取网络请求失败情况、获取页面加载性能指标、获取页面内容/HTML/链接/表单
- **可视化类**：全页截图、元素截图、带标注的交互快照、响应式多尺寸截图、页面内容 diff、PDF 导出

### 4. 智能用户接管机制
完美解决自动化测试的常见卡点：
- 遇到 CAPTCHA、人机验证、多因素认证（MFA）、复杂 OAuth 登录这些 AI 搞不定的场景，自动打开可见的 Chrome 浏览器，把控制权交给用户
- 用户操作完成后，输入 resume 命令就能自动拿回控制权，继续执行后续操作
- 浏览器状态完全保留，用户操作后的 Cookie、登录状态都会继承，不用重新操作

### 5. 多场景支持能力
- **多标签页**：支持新建、切换、关闭标签页，模拟多页面操作场景
- **iframe 支持**：切换到 iframe 上下文操作，支持嵌套 iframe 的复杂页面
- **跨环境对比**：支持 diff 两个 URL 的页面内容，对比 staging 和生产环境的差异
- **状态持久化**：保存和加载浏览器状态，下次启动直接恢复登录状态和页面，不用重新登录
- **响应式测试**：自动生成移动端、平板、桌面端三个尺寸的截图，一键测试响应式布局

### 6. 丰富的验证能力
不用写断言代码，直接用内置命令验证：
- `is visible ".dashboard"`检查元素是否可见
- `console --errors`检查有没有 JS 错误
- `network`检查有没有失败的网络请求
- `perf`检查页面加载性能指标
- `text`检查页面有没有预期的内容
- `js "document.title.includes('Dashboard')"`执行自定义 JS 表达式验证

### 7. Bug 证据自动收集
复现 Bug 的时候自动收集所有证据：
- 带标注的截图，清晰展示问题位置
- 控制台错误日志，定位 JS 问题
- 网络请求列表，定位接口错误
- 页面内容和 HTML，方便后续分析
- 所有证据可以直接导出，不用手动截图复制日志

## 五、典型使用场景

- 自动化测试用户流程：登录、注册、下单、提交表单等常见用户流程
- 上线后快速验证页面功能是否正常，有没有控制台错误、接口失败
- 复现线上 Bug，自动收集截图、错误日志、网络请求等证据，方便排查
- 测试响应式布局在移动端、平板、桌面端的显示是否正常
- 测试文件上传、弹窗处理、多步骤流程等复杂交互功能
- 对比 staging 和生产环境的页面差异，确保上线内容正确
- 自动收集页面性能指标，监控性能变化
- 作为其他测试角色（/qa、/canary、/design-review）的底层工具，提供浏览器操作能力

## 六、实际使用示例

> **用户输入**：帮我测试下登录流程是否正常，用户名为 user@test.com，密码为 123456。
>
> **/browse 输出**：
> 启动无头浏览器，跳转到登录页 https://xxx.com/login
> 运行`snapshot -i -a`获取交互元素：
> @e1 [textbox] "邮箱/用户名"
> @e2 [textbox] "密码"
> @e3 [button] "登录"
> @e4 [link] "忘记密码"
> 截图已保存为/tmp/login-annotated.png
>
> 执行：
> fill @e1 "user@test.com"
> fill @e2 "123456"
> click @e3
> wait ".dashboard"
>
> 验证结果：
> 页面成功跳转到/dashboard，.dashboard 元素可见
> 控制台检查：0 个错误
> 网络请求检查：所有接口请求成功
> 页面加载时间：850ms
> 登录成功后的截图已保存为/tmp/dashboard.png
>
> ✅ 登录流程测试通过，功能正常。

---

> **用户输入**：遇到 CAPTCHA 验证了，过不去。
>
> **/browse 输出**：
> 自动打开可见 Chrome 浏览器，当前页面是登录页的 CAPTCHA 验证，请手动完成验证，完成后输入 resume 继续。
>
> （用户完成验证后输入 resume）
> 拿回控制权，当前页面已经跳转到仪表盘，登录流程完成，后续操作可以继续执行。

## 七、迁移价值

这个角色的设计思路完全可以脱离 Gstack 复用：

- 极简 API 的无头浏览器设计思路可以复用在团队的自动化测试工具里，不用写复杂的 Puppeteer/Playwright 脚本，简单的命令就能完成测试，降低自动化测试的门槛，让没有测试经验的开发者也能写自动化用例
- 自动元素编号引用的方式可以大大降低选择元素的难度，不用写复杂的 CSS 选择器和 XPath，看元素编号就能操作，提升测试脚本的编写效率
- 用户接管机制可以解决自动化测试遇到人机验证、多因素认证、复杂 OAuth 的痛点，不用因为这些卡点放弃整个自动化流程，兼顾自动化的效率和手动操作的灵活性
- 极快的运行速度可以大大提升测试用例的执行效率，让测试反馈更快，提升开发效率
- 核心逻辑是「工具要简单好用，解决 90% 的常见场景，剩下 10% 开放出口给用户」，适用于所有效率工具的设计，让普通用户也能用上复杂的底层能力。
