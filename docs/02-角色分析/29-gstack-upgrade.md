# /gstack-upgrade：版本管理员

## 一、核心定位

`/gstack-upgrade` 是 Gstack 自身版本管理的专属角色，定位是一键升级器，核心目标是帮助用户轻松将 Gstack 升级到最新版本，自动适配全局安装、项目级 vendored 安装、Git 安装等不同的部署方式，自动处理备份、升级、回滚、本地副本同步全流程，升级后智能展示用户可见的新功能和优化点，支持自动升级、延迟提醒、永不提醒多种灵活选项，让用户可以低成本始终使用最新版本的功能、bug 修复和安全更新，不用手动下载替换文件，简化版本维护成本。

它是 Gstack 自身迭代的核心工具，解决了 CLI 工具升级麻烦、用户容易停留在旧版本错过新功能的痛点，让所有用户都能轻松用上最新版本。

## 二、设计理念

这个技能的设计完全继承了"无感自动升级降低维护成本"和"用户选择权优先"的最佳实践，背后的核心逻辑是：

> 大部分用户不会主动升级 CLI 工具，要么觉得升级麻烦，要么不知道有新版本，导致很多用户停留在旧版本，错过新功能、bug 修复和安全更新。把升级流程完全自动化，一键完成所有操作，升级失败还能自动回滚，用户没有后顾之忧；同时提供灵活的提醒选项，想要最新功能的可以开自动升级，不想被打扰的可以关闭提醒，兼顾不同用户的偏好，大大提升用户使用最新版本的比例。

`/gstack-upgrade` 有两条核心原则：1. 安全第一，升级失败自动回滚到旧版本，绝对不会导致工具不可用；2. 用户选择权优先，提供多种升级和提醒选项，不强制用户升级，也不会频繁骚扰用户。

## 三、标准工作流程

调用 `/gstack-upgrade` 或者检测到新版本自动触发时，它会按照标准化的流程完成升级：

1. **升级策略检查阶段**：首先检查是否开启了自动升级，如果配置了自动升级，直接进入升级流程；否则弹出选项询问用户：立即升级、始终自动升级、稍后提醒、永不提醒。
2. **用户选择处理阶段**：
   - 立即升级：进入升级流程
   - 始终自动升级：配置自动升级开关，后续新版本自动安装
   - 稍后提醒：按照递增退避策略延迟提醒，第一次 24 小时，第二次 48 小时，之后每周提醒一次
   - 永不提醒：关闭版本更新检查，不再提示任何升级信息
3. **安装类型检测阶段**：自动检测 Gstack 的安装类型和路径：
   - 全局 Git 安装：安装在用户主目录的全局路径
   - 本地 Git 安装：项目本地的 Git 克隆安装
   - Vendored 安装：项目内复制的 vendored 副本
   - 全局 Vendored 安装：全局的 vendored 部署
4. **升级执行阶段**：
   - 备份旧版本，防止升级失败
   - Git 安装类型：stash 本地修改，拉取最新代码，重置到 main 分支，运行 setup 脚本
   - Vendored 安装类型：临时目录克隆最新版本，替换旧版本，运行 setup 脚本
   - 升级失败自动恢复备份的旧版本，提示用户升级失败，不影响现有使用
5. **本地副本同步阶段**：如果检测到当前项目里有本地 vendored 的 Gstack 副本，自动同步升级到最新版本，提示用户提交变更。
6. **更新缓存清理阶段**：清除更新检查缓存、提醒 snooze 标记，确保后续版本检测正常。
7. **更新内容展示阶段**：读取 CHANGELOG.md，智能提取旧版本和新版本之间的用户可见的新功能、重要修复，生成 5-7 条摘要展示给用户，忽略内部重构内容。
8. **流程恢复阶段**：如果是在其他技能的前置检测中触发的升级，升级完成后继续执行原来的技能流程，用户感知不到中断。

## 四、核心能力

### 1. 多安装类型全适配
支持 Gstack 所有常见的部署方式，不需要用户手动指定安装路径：
- 全局 Git 安装：用户主目录下的全局安装
- 本地 Git 安装：项目内单独克隆的 Git 仓库安装
- 项目级 Vendored 安装：复制到项目.claude/skills 目录下的 vendored 副本
- 全局 Vendored 安装：全局路径下的 vendored 部署
- 自动识别安装类型和路径，执行对应的升级逻辑，适配所有使用场景

### 2. 灵活的升级策略
满足不同用户的偏好：
- **自动升级**：配置后新版本自动安装，不需要用户干预，始终保持最新
- **手动确认升级**：每次新版本弹出提示，用户确认后升级
- **延迟提醒**：暂时不想升级可以选择稍后提醒，按照退避策略延迟，不会频繁骚扰
- **永不提醒**：关闭版本检测，永远不会收到升级提示，适合追求稳定不想变动的用户

### 3. 安全可靠的升级保障
- 升级前自动备份旧版本，升级过程中出现任何错误自动回滚到旧版本，绝对不会导致工具不可用
- 本地修改自动 stash 保存，升级后提示用户可以恢复，不会丢失用户的本地修改
- 升级过程全程无感知，自动升级不会打断用户的正常工作，升级完成后才提示更新内容

### 4. 本地副本自动同步
- 自动检测当前项目里的 vendored Gstack 副本，如果存在自动同步升级到最新版本
- 不用用户手动更新每个项目的 vendored 副本，一次全局升级所有项目都受益
- 同步失败自动恢复本地副本，不会破坏项目内的现有配置

### 5. 智能更新内容摘要
- 自动提取两个版本之间的 CHANGELOG 内容，只展示用户可见的新功能、重要修复、安全更新
- 忽略内部重构、代码优化这些用户不关心的内容，重点突出对用户有用的变化
- 简洁明了，不用用户自己翻长文档找更新内容

### 6. 无缝集成升级
- 可以在其他技能的前置检测中自动触发升级，升级完成后自动继续原来的流程，用户几乎感知不到
- 不需要用户专门运行升级命令，使用过程中就能自动完成升级
- 不影响用户的正常工作流

## 五、典型使用场景

- 收到新版本升级提示的时候，一键升级到最新版本
- 想体验最新发布的功能的时候，主动运行升级
- 遇到已知 bug，升级看看新版本有没有修复
- 开启自动升级，始终保持最新版本，不用手动管理
- 多项目使用 vendored 安装 Gstack 的时候，一次全局升级同步所有项目的副本
- 新版本包含安全更新的时候，快速升级避免安全风险

## 六、实际使用示例

> **用户输入**：/gstack-upgrade
>
> **/gstack-upgrade 输出**：
> 检测到当前版本：v1.0.0，最新版本：v1.1.0，是否升级？
> A) 立即升级 B) 始终自动升级 C) 稍后提醒 D) 永不提醒
>
> （用户选择 A）
> 检测安装类型：全局 Git 安装，路径：~/.claude/skills/gstack
> 备份旧版本完成，开始升级...
> 拉取最新代码，重置到 origin/main，运行 setup 脚本完成。
> 检测到当前项目有本地 vendored 副本：./.claude/skills/gstack，已同步升级到 v1.1.0，请记得提交变更。
>
> ✅ 升级成功：v1.0.0 → v1.1.0
>
> 新功能：
> - 新增 /setup-deploy 部署配置工具，一次配置永久自动部署
> - 新增 /connect-chrome 可见浏览器控制，支持侧边栏实时操作和日志
> - 优化 /qa 测试速度，提升 300%
> - 修复 /review 评审遗漏安全问题的 bug
> - 新增自动升级功能，后续版本可以自动安装
>
> Happy shipping!

> **场景：自动升级触发**

> 日志输出：Auto-upgrading gstack v1.0.0 → v1.1.0...
> 升级完成后自动继续/ship 的流程，最后提示升级完成和新功能。
> 用户感知不到升级中断，正常完成发布流程。

## 七、迁移价值

这个角色的设计思路完全可以复用在 CLI 工具的版本管理里：

- 自动升级+多安装类型适配的思路可以大大降低用户的升级成本，提升用户使用最新版本的比例，避免大量用户停留在旧版本
- 灵活的提醒选项可以平衡升级提示和用户体验，不会造成骚扰，不同偏好的用户都能接受
- 自动备份回滚机制消除了用户的升级顾虑，不用担心升级失败导致工具不可用，用户更愿意升级
- 本地 vendored 副本自动同步的能力适合工具在多项目里复用的场景，一次升级所有项目都受益，不用每个项目单独升级
- 无感集成升级的方式不会打断用户的正常工作流，升级过程对用户透明，体验更好
- 核心逻辑是「升级流程自动化、安全、无感，降低用户升级的门槛和顾虑，让用户轻松保持最新版本」，适用于所有需要版本管理的 CLI 工具和桌面软件。
