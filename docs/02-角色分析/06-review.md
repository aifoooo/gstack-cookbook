# /review：资深工程师

## 一、核心定位

`/review` 是 Gstack 代码质量的最后一道防线，完全复刻了资深工程师的代码评审逻辑，它的定位就是代码质量的首席把关人，核心目标是在代码合并上线前，找出所有测试覆盖不到的结构性问题、安全漏洞、性能隐患，避免线上故障和技术债务。

它是代码上线前的强制评审环节，所有需要合并到主分支的代码都必须经过它的评审，是保障代码质量的核心保障。

## 二、设计理念

这个技能的设计完全继承了硅谷一线科技公司的代码评审最佳实践，背后的核心逻辑是：

> 再好的自动化测试也覆盖不到所有的场景，代码评审是捕捉结构问题、安全漏洞、性能隐患的最佳时机。资深工程师的评审经验可以避免很多低级错误和线上故障，这些问题一旦上线，修复成本是上线前的100倍。

`/review` 内置了完整的代码评审检查清单，覆盖安全、并发、性能、测试、设计等全维度问题，同时支持多模型对抗评审，diff越大评审越严格，确保不漏过任何严重问题。

## 三、标准工作流程

调用 `/review` 后，它会按照标准化的流程开展代码评审，不会跳过任何关键环节：

1. **前置分支检查**：先确认当前分支有没有和主分支的差异，如果在主分支或者没有变更就直接结束
2. **范围漂移检查**：对比需求和实际变更，检查有没有做计划外的功能（范围蔓延），有没有漏掉需求（需求缺失），确保代码变更符合预期
3. **多轮次深度评审**：分两轮评审：
   - 第一轮（严重问题）：SQL安全、并发竞争条件、LLM输出信任边界、枚举值完整性，这些问题会直接导致线上故障，优先检查
   - 第二轮（普通问题）：条件副作用、魔法数字、死代码、LLM提示词问题、测试缺口、前端设计问题、性能和包体积影响
4. **测试覆盖度审计**：自动分析变更的代码路径，生成覆盖度图表，找出所有未覆盖的测试缺口，包括代码分支和用户流程
5. **自动修复简单问题**：对于机械性的简单问题（比如CSS错误、代码格式、明显的小bug）直接自动修复，不用用户手动改
6. **批量确认复杂问题**：对于需要用户决策的问题（比如架构问题、安全漏洞）批量展示，说明问题和修复建议，让用户确认是否修复
7. **可选对抗性评审**：根据diff大小自动调整严格程度：小diff跳过，中等diff启用单模型对抗评审，大diff启用多模型交叉评审，确保大变更的质量
8. **输出评审报告**：生成标准化的评审报告，包含发现的问题、修复情况、测试覆盖度、下一步建议
9. **自动衔接后续流程**：评审通过后可以直接调用 `/ship` 上线，不通过的话修复后重新评审

## 四、核心能力

### 1. 全维度代码问题识别能力
覆盖代码评审的所有核心维度，能识别：
- 安全问题：SQL注入、XSS、权限漏洞、敏感数据泄露
- 并发问题：竞态条件、死锁、资源泄露
- 数据问题：类型错误、枚举值不完整、数据不一致
- LLM相关问题：提示词注入、输出未校验就写入数据库
- 测试问题：测试覆盖缺口、回归风险
- 前端问题：不符合设计规范、无障碍问题、响应式问题
- 性能问题：N+1查询、内存泄露、包体积过大
- 工程问题：魔法数字、死代码、代码重复、不遵循规范

### 2. 自动修复简单问题
对于不需要决策的机械性问题直接自动修复，比如：
- 代码格式问题
- 简单的CSS错误（比如outline:none、字体太小）
- 明显的语法错误
- 遗漏的简单校验
- 自动补充简单的单元测试

### 3. 范围漂移自动检测
自动对比需求文档、PR描述、提交记录和实际代码变更，识别：
- 范围蔓延：做了需求之外的功能，增加不必要的复杂度
- 需求缺失：没有完成承诺的功能，漏掉了核心逻辑
- 计划变更：实现方式和计划不一致，提前和用户确认是否是预期变更

### 4. 测试覆盖度智能审计
不仅仅统计行覆盖率，而是从代码路径和用户流程两个维度分析：
- 所有代码分支的覆盖情况（if/else、异常分支、边界条件）
- 所有用户交互流程的覆盖情况（正常流程、异常场景、边缘操作）
- 自动区分哪些需要单元测试，哪些需要E2E测试，哪些需要LLM效果评估
- 覆盖率低于阈值自动预警

### 5. 多模型对抗评审
根据diff大小自动调整评审严格程度：
- 小diff（<50行）：正常评审即可
- 中等diff（50-199行）：启用额外的模型做对抗性评审，找主模型漏过的问题
- 大diff（>=200行）：启用四轮交叉评审（Claude结构化+Claude对抗+Codex结构化+Codex对抗），确保大变更没有严重问题
- 高风险变更（支付、权限、数据相关）自动提升评审级别

### 6. 历史经验学习
自动学习同一项目甚至跨项目的历史踩坑记录，避免重复出现同样的问题，评审标准会随着使用越来越贴合团队的技术栈和实际情况。

## 五、典型使用场景

- 代码合并前的PR评审，确保代码质量达标
- 写完代码后的自查，提前发现问题不用等同事评审
- 重构后的代码质量检查，确保重构没有引入新问题
- 新人提交代码的评审，统一代码风格和质量标准
- 安全敏感功能（支付、权限、数据）的代码评审，确保没有安全漏洞
- 大版本上线前的整体代码质量审计

## 七、迁移价值

这个角色的设计思路完全可以脱离 Gstack 复用：

- 你可以把这套代码评审流程用到自己团队的代码评审环节，任何代码合并前都按照这个流程走一遍
- 内置的检查清单可以直接作为团队的代码评审标准，统一评审口径
- 范围漂移检查、测试覆盖度审计、多模型对抗评审这些方法都可以直接复用
- 核心逻辑是「代码合并前做全维度评审，把问题消灭在上线前」，适用于所有软件研发场景
