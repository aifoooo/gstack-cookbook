# /investigate：调试专家

## 一、核心定位

`/investigate` 是 Gstack 问题排查的核心角色，完全复刻了资深运维/开发工程师的系统性故障排查逻辑，它的定位就是线上问题的首席侦探，核心目标是彻底找到问题的根本原因，而不是只修复表面症状，避免"打地鼠"式的调试，确保问题被永久解决不会反复出现。

它是所有问题排查场景的入口，遇到报错、500错误、异常行为、"昨天还好好的今天就坏了"这类问题时都会自动调用，是保障线上稳定的核心角色。

## 二、设计理念

这个技能的设计完全继承了Google SRE的故障排查最佳实践，背后的核心逻辑是：

> 没有找到根本原因的修复都是在制造更多问题。很多团队调试Bug都是"症状好了就行"，但这样的修复会埋下更多隐患，导致后续问题越来越难排查，最终付出10倍的代价。调试的核心不是修复症状，而是找到根因一劳永逸地解决。

`/investigate` 有一条不可动摇的铁律：**没有根因分析就绝对不做任何修复**，完全遵循"调查→分析→假设→验证→修复→回归"的科学排查流程，杜绝猜谜式调试。

## 三、标准工作流程

调用 `/investigate` 后，它会按照标准化的流程系统性排查问题，不会跳过任何关键环节：

1. **根因调查阶段**：先收集所有症状，包括错误信息、栈追踪、复现步骤，追溯代码路径，查看最近的变更记录，尝试稳定复现问题，形成具体的根因假设
2. **排查范围锁定**：自动锁定问题涉及的最小目录范围，禁止修改无关代码，防止范围蔓延和不必要的重构
3. **模式分析阶段**：匹配已知问题模式（竞态条件、空指针、状态损坏、集成失败、配置漂移、缓存失效等），搜索同类型问题的已知解决方案
4. **假设验证阶段**：验证根因假设，如果3次假设都不成立就自动升级，避免在错误的方向上浪费时间
5. **修复实现阶段**：只做最小改动修复根因，不修改无关代码，自动编写回归测试，确保同样的问题不会再出现
6. **验证报告阶段**：复现原问题场景确认修复有效，运行全量测试确保没有引入回归，输出结构化的调试报告，包含症状、根因、修复、证据、回归测试等完整信息
7. **自动衔接后续流程**：修复完成后可以直接调用 `/review` 做代码评审，再通过 `/ship` 上线

## 四、核心能力

### 1. 系统性根因排查能力
完全遵循科学排查流程，不会像普通调试一样东改西改：
- 先收集完整信息再做假设，不会上来就瞎改代码
- 追溯问题的完整上下文，包括最近的变更、历史相关问题、外部依赖状态
- 必须稳定复现问题才会进入修复阶段，避免误判

### 2. 已知问题模式匹配
内置常见问题模式库，可以快速定位：
- 竞态条件：间歇性、时间相关的问题
- 空指针传播：方法不存在、类型错误类问题
- 状态损坏：数据不一致、部分更新类问题
- 集成失败：超时、外部服务返回异常类问题
- 配置漂移：本地正常、线上/ staging 失败类问题
- 缓存失效：显示旧数据、清缓存就好类问题

### 3. 排查范围自动锁定
自动识别问题涉及的最小代码范围，锁定后只能修改这个范围内的代码，避免：
- 调试的时候顺手改了一堆无关的代码，引入更多问题
- 把简单的bug修复变成了不必要的重构
- 改到了其他模块的代码导致回归

### 4. 回归测试自动生成
修复根因后自动生成对应的回归测试：
- 没有修复的时候测试是失败的，证明测试的有效性
- 修复后测试是通过的，证明修复真的解决了问题
- 确保同样的问题以后不会再出现

### 5. 历史经验学习
自动学习同一项目甚至跨项目的历史问题记录，遇到同类问题可以快速定位，避免重复踩同样的坑，排查效率会随着使用越来越高。

### 6. 外部问题自动搜索
遇到未知的问题模式会自动搜索：
- 框架、库的已知问题和解决方案
- 同类型错误的行业最佳实践
- 官方文档和社区的已知bug报告

## 五、典型使用场景

- 线上bug排查，需要快速定位根因的时候
- "昨天还好好的今天就坏了"的回归问题排查
- 间歇性、偶现的疑难杂症排查
- 报错信息不明确，不知道哪里出问题的时候
- 问题涉及多个模块，不知道从何下手的时候
- 修复后反复出现的问题，需要彻底解决的时候

## 六、实际使用示例

> **用户输入**：支付回调有时候会失败，没有规律，帮我排查下。
>
> **/investigate 输出**：
> 我先收集了症状：支付回调接口偶尔返回500，没有固定规律，上周上线支付功能后出现的。
>
> 我查看了最近的变更记录：上周三提交的支付回调代码里，网络请求没有超时重试机制，而且幂等校验用的是内存锁，分布式场景下会失效。
>
> 根因假设：当第三方支付网关响应超时的时候，代码没有重试直接抛出异常，同时并发回调的时候内存锁失效会导致重复处理数据出错。
>
> 我已经锁定排查范围为 `src/services/pay/` 目录，不会修改其他代码。
>
> 我验证了假设：模拟超时场景确实会出现500错误，并发场景下确实会出现重复处理的问题，根因确认。
>
> 修复方案：
> 1. 给第三方接口调用加3次超时重试机制
> 2. 把内存幂等锁改成数据库唯一索引+分布式锁
> 3. 自动生成了对应的回归测试，覆盖超时和并发场景
>
> 验证结果：原错误场景已经可以稳定复现，修复后不再出现，全量测试通过，没有引入回归。
>
> 调试报告：
> ```
> Symptom:         支付回调偶现500错误
> Root cause:      缺少超时重试机制+分布式场景下幂等校验失效
> Fix:             src/services/pay/callback.js:42 加重试逻辑，src/services/pay/idempotent.js:18 改分布式锁
> Evidence:        模拟100次超时场景，修复前失败37次，修复后全部成功
> Regression test: test/services/pay/callback.test.js:89，覆盖超时和并发场景
> Status:          DONE
> ```

## 七、迁移价值

这个角色的设计思路完全可以脱离 Gstack 复用：

- 你可以把这套系统性排查流程用到自己团队的故障排查规范里，任何线上问题都按照这个流程走一遍
- "没有根因就不修复"、"3次假设失败就升级"、"最小改动修复"这些铁律可以直接作为团队的调试准则
- 内置的问题模式库可以直接作为团队的故障排查手册，新人也能快速定位常见问题
- 核心逻辑是「先找到根因再修复，而不是修复表面症状」，适用于所有软件故障排查场景
