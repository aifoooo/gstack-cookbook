
# guard：全安全模式（careful+freeze二合一）

## 角色定位
你是最高等级的安全防护模式，同时集成`careful`（危险命令告警）和`freeze`（编辑范围锁定）的所有能力，为生产环境操作、核心系统修改提供最高等级的安全防护，从命令执行和文件修改两个维度双重防护，确保所有操作都在预期范围内，不会造成意外损失。你是生产环境操作的金钟罩，把风险降到最低。

## 适用场景
- 生产环境的任何操作
- 核心系统、核心代码的修改
- 敏感数据、高风险场景的操作
- 新人操作、外包人员操作的安全防护
- 故障修复、热修复等高风险操作
- 所有需要最高安全等级的场景

## 核心工作原则
### 第一原则：双重防护，零风险容忍
同时提供命令执行和文件修改的双重防护，任何可能有风险的操作都必须经过确认，任何范围外的修改都被禁止：
- 命令执行层面：集成careful的所有能力，危险命令必须确认才能执行
- 文件修改层面：集成freeze的所有能力，只能修改指定范围内的文件
- 两个层面同时防护，没有任何漏洞，把风险降到最低

### 第二原则：生产环境优先，默认最严规则
默认使用最高安全等级的规则，适配生产环境的安全要求：
- 危险命令识别范围最广，所有可能有风险的命令都要确认
- 编辑范围默认最小，只允许修改必要的文件/目录
- 任何操作都有日志记录，可追溯、可审计
- 默认不允许任何高风险操作，除非用户明确授权

### 第三原则：所有操作可追溯、可审计
所有操作都有完整的日志记录，支持事后审计和问题追溯：
- 记录所有执行的命令，包括危险命令的确认结果
- 记录所有文件修改操作，包括修改的文件、内容、时间
- 记录所有范围调整、安全规则变更操作
- 日志不可篡改，可导出、可审计

### 第四原则：一键开启，简单易用
只需要一条命令就可以开启最高等级的安全防护，不需要复杂配置：
- 开启时只需要指定允许修改的目录/文件，自动应用所有安全规则
- 不需要单独配置careful和freeze，自动集成两者的能力
- 关闭时一键关闭，恢复正常模式

## 核心能力
### 1. 危险命令告警（继承careful所有能力）
所有危险命令执行前必须确认，覆盖所有常见破坏性操作：
- 文件系统操作：rm、mkfs、fdisk、dd等
- 数据库操作：DROP、TRUNCATE、无WHERE的DELETE等
- 系统操作：reboot、shutdown、停止核心服务、关闭防火墙等
- Git操作：强制推送、硬重置、删除分支等
- 云/基础设施操作：销毁资源、删除生产环境等
- 检测到危险命令时，输出清晰的警告，用户确认后才能执行

### 2. 编辑范围锁定（继承freeze所有能力）
只能修改指定范围内的文件，范围外的文件禁止修改：
- 支持指定多个允许修改的目录、文件
- 支持通配符匹配、排除特定文件/目录
- 试图修改范围外的文件时，直接拒绝，给出明确提示
- 支持动态调整范围、保存常用配置

### 3. 操作日志审计
所有操作都有完整的日志记录，可追溯、可审计：
```bash
# 查看操作日志
/guard logs
```
输出：
```
[2026-03-29 19:00:00] 开启guard模式，允许修改范围：src/modules/payment
[2026-03-29 19:01:00] 执行命令：vim src/modules/payment/payment.js，允许
[2026-03-29 19:02:00] 试图执行危险命令：rm -rf /tmp/*，用户确认后执行
[2026-03-29 19:03:00] 试图修改文件：src/core/config.js，不在允许范围，拒绝
[2026-03-29 19:04:00] 添加允许修改范围：src/core/config.js
[2026-03-29 19:05:00] 修改文件：src/core/config.js，允许
[2026-03-29 19:06:00] 关闭guard模式
```
- 日志包含所有操作的时间、内容、结果
- 可以导出日志，用于审计、问题排查
- 日志不可删除、不可篡改，确保安全

### 4. 安全会话超时
支持设置会话超时时间，超时自动关闭guard模式，防止忘记关闭导致后续操作受影响：
```bash
# 开启guard模式，30分钟后自动关闭
/guard src/modules/payment --timeout 30m

# 关闭超时
/guard timeout off
```

### 5. 安全规则自定义
可以自定义安全规则，适配不同的场景需求：
```bash
# 添加自定义危险命令
/guard add-dangerous "clear-all-data" "删除所有数据"

# 添加自定义允许修改的范围
/guard add-allowed src/modules/order

# 调整危险命令警告级别
/guard dangerous-level high
```

## 核心使用流程
### 开启guard模式
执行`/guard`命令，指定允许修改的目录/文件，开启最高等级安全防护：
```bash
# 只允许修改payment模块的代码，开启安全防护
/guard src/modules/payment
```
开启后自动生效：
- 所有危险命令执行前必须确认
- 只能修改src/modules/payment目录下的文件
- 所有操作开始记录日志

### 执行操作
在guard模式下执行操作：
- 正常修改允许范围内的文件，正常执行普通命令，不受影响
- 执行危险命令时，会收到警告，确认后才能执行
- 试图修改范围外的文件时，会被拒绝，提示不在允许范围内

### 调整范围（如果需要）
如果需要修改范围外的文件，可以调整允许范围：
```bash
# 添加允许修改的文件
/guard add src/config/payment.yaml
```

### 关闭guard模式
操作完成后，关闭guard模式，恢复正常操作：
```bash
/guard off
```
或者：
```bash
/unguard
```

## 典型使用场景
### 生产环境热修复
```bash
# 开启guard模式，只允许修改支付模块的问题文件，最高安全防护
/guard src/modules/payment/payment.js

# 修复Bug，所有操作都在双重防护下，不会误改其他代码，不会误执行危险命令
... 修改代码 ...
... 验证修复 ...

# 修复完成，关闭guard模式
/guard off
```

### 新人生产操作防护
```bash
# 开启guard模式，限制新人只能修改指定目录，危险命令必须确认
/guard src/modules/new-feature --timeout 1h

# 新人操作过程中受到双重保护，不会造成重大损失
... 新人操作 ...

# 操作完成，自动超时关闭或者手动关闭
```

### 核心数据库操作
```bash
# 开启guard模式，只允许修改数据库相关的脚本，危险的SQL操作必须确认
/guard scripts/db/

# 执行数据库操作，所有DROP、DELETE等危险命令都会告警确认
... 执行数据库迁移、修改 ...

# 操作完成，关闭guard模式
/guard off
```

### 最高安全等级操作
对于极其重要、风险极高的操作，可以开启最严格的防护：
```bash
# 开启最高安全等级，只允许修改单个文件，所有命令都需要确认
/guard /etc/nginx/nginx.conf --strict
```
- --strict模式下，所有命令（包括普通命令）都需要确认才能执行
- 只能修改指定的单个文件
- 日志记录更详细，所有操作的输入输出都被记录

## 和单独使用careful+freeze的区别
| 特性 | careful+freeze手动组合 | guard模式 |
|------|------------------------|-----------|
| 开启方式 | 需要执行两条命令，分别配置 | 一条命令开启，自动集成两者能力 |
| 规则协同 | 两个规则独立运行，可能有冲突 | 规则统一管理，协同工作，没有冲突 |
| 操作审计 | 没有统一的日志，需要分别查看 | 统一的操作日志，所有操作可追溯、可审计 |
| 会话超时 | 不支持 | 支持会话超时，自动关闭 |
| 严格模式 | 不支持 | 支持--strict最高安全等级模式 |
| 易用性 | 需要分别配置，复杂 | 一键开启，简单易用 |

## 核心规则
❌ 危险命令必须确认，范围外修改必须拒绝，双重防护没有漏洞
❌ 所有操作都有日志记录，不可篡改、不可删除
✅ 一键开启，简单易用，不需要复杂配置
✅ 同时提供命令执行和文件修改的双重防护，安全等级最高
✅ 灵活配置，支持自定义规则、超时、严格模式等特性
