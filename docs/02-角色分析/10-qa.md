# /qa：测试负责人

## 一、核心定位

`/qa` 是 Gstack 上线前功能质量的最后一道防线，完全复刻了资深测试工程师+开发工程师的一体化测试修复逻辑，它的定位就是产品质量的首席测试官，核心目标是像真实用户一样完整测试产品的所有功能，发现 bug 后直接修复，不需要转给开发，实现"测试→修复→验证"全流程一站式完成，大大缩短测试修复的周期。

和静态代码评审不同，它是完全站在用户角度做黑盒测试，真实点击每个按钮、填写每个表单、走通每个流程，覆盖所有自动化测试覆盖不到的真实用户场景，是上线前最接近真实用户体验的测试环节。

## 二、设计理念

这个技能的设计完全继承了 Google "测试左移"和"全栈工程师"的最佳实践，背后的核心逻辑是：

> 传统测试流程里测试人员只找 bug，再转给开发修复，沟通成本高，周期长，很多小 bug 因为流程繁琐甚至不会被修复。AI 可以同时具备测试和开发能力，发现问题直接修复，整个流程从几天缩短到几分钟，而且测试要像真实用户一样操作，而不是只跑预设的自动化用例，才能覆盖到用户真实会遇到的所有场景。

`/qa` 有三个核心原则：1）所有问题必须有复现证据（截图/录屏）；2）能修复的问题直接修复，不甩锅；3）测试要完全模拟真实用户行为，不看源代码。

## 三、标准工作流程

调用 `/qa` 后，它会按照标准化的流程开展测试和修复，不会跳过任何关键环节：

1. **前置准备阶段**：确认测试目标 URL、测试层级（快速/标准/全面）、测试范围、认证信息，如果是在功能分支上会自动进入 diff 感知模式，只测试代码改动影响到的页面和功能，不用全量测试节省时间
2. **环境准备**：自动检测本地运行的服务，导入用户 cookie 或者自动登录，准备测试所需的权限
3. **应用地图探索**：先访问首页，爬取所有可访问的页面和导航路径，形成测试地图，优先测试核心功能路径
4. **逐页测试阶段**：按照测试地图逐个页面测试，每个页面执行完整的检查清单：视觉扫描、交互元素测试、表单测试、导航测试、状态测试（空/加载/错误）、控制台错误检查、响应式测试
5. **问题实时记录**：发现问题立刻记录证据，包括截图、复现步骤、错误信息，每个问题都有可复现的完整证据
6. **问题分级修复**：按严重程度（致命/高/中/低）排序，逐个修复问题，每个修复单独原子提交，修复后立刻重新验证，生成修复前后的对比截图
7. **回归测试自动生成**：修复的 bug 自动生成对应的回归测试用例，确保同样的问题不会再出现
8. **最终验证阶段**：所有问题修复完后重新跑一遍完整测试，计算最终的健康评分，和基线做对比
9. **测试报告输出**：输出完整的测试报告，包含发现的问题、修复情况、前后健康分对比、所有证据截图、后续建议

## 四、核心能力

### 1. 多模式灵活测试能力
支持多种测试场景：
- **diff 感知模式**：功能分支自动识别代码改动，只测试影响到的页面和功能，最常用的模式，节省大量时间
- **全量模式**：全站完整测试，覆盖所有可访问页面
- **快速模式**：30 秒冒烟测试，只测首页+核心 5 个页面，看基本功能是否正常
- **回归模式**：和之前的测试基线对比，看哪些问题修好了，哪些是新出现的问题

### 2. 全维度问题检测能力
覆盖 8 大类测试维度，加权计算健康总分：
- **控制台错误**（权重 15%）：JS 错误、接口报错等
- **链接有效性**（权重 10%）：死链、404 错误等
- **视觉问题**（权重 10%）：布局错乱、样式错误、显示异常等
- **功能问题**（权重 20%）：按钮失效、表单提交失败、流程走不通等
- **体验问题**（权重 15%）：交互卡顿、反馈不及时、操作反人类等
- **性能问题**（权重 10%）：加载慢、卡顿、资源过大等
- **内容问题**（权重 5%）：错别字、文案错误、空状态提示不合理等
- **可访问性问题**（权重 15%）：对比度不够、键盘导航不支持、屏幕阅读器适配问题等

### 3. 自动修复能力
大部分问题可以直接自动修复：
- 前端 JS 逻辑错误、样式问题、文案错误
- 简单的后端逻辑错误、参数校验问题
- 每个修复单独原子提交，方便回溯和回滚
- 修复后立刻重新验证，确保真的解决了问题

### 4. 回归测试自动生成能力
修复的 bug 会自动生成对应的回归测试：
- 自动学习项目的测试规范和风格，生成的测试和项目已有测试保持一致
- 覆盖 bug 的触发条件和修复后的验证逻辑
- 确保同样的问题不会再次出现

### 5. 多框架专项适配能力
针对主流框架做专项检查：
- **Next.js**：检查水合错误、客户端路由问题、数据请求错误
- **Rails**：检查 N+1 查询、CSRF token 问题、Turbo 交互问题
- **WordPress**：检查插件冲突、REST API 问题、混合内容警告
- **SPA 应用**：检查客户端路由、状态管理、浏览器前进后退问题

### 6. 量化健康评分体系
用 0-100 分量化网站质量，每个维度都有明确的打分标准，测试前后分数对比直观体现质量提升情况，每次测试的基线会永久保存，可以对比不同版本的质量变化。

## 五、典型使用场景

- 新功能开发完成后，上线前的完整测试
- 功能分支代码合并前的测试验证
- 线上问题修复后的回归测试
- 定期全站健康检查，及时发现隐性问题
- 没有专业测试团队的小团队做产品质量保障
- 大版本上线前的全面功能测试

## 六、实际使用示例

> **用户输入**：我刚写完支付功能，在 feature/pay 分支，帮我测试下有没有问题。
>
> **/qa 输出**：
> 我自动识别到是 feature/pay 功能分支，分析代码改动影响了支付页、支付结果页、订单列表页三个页面，检测到本地服务运行在 http://localhost:3000。
>
> 开始测试支付相关流程，共发现 3 个问题：
> ### 严重问题（1 个）
> 1. 支付表单没有做非空校验，用户不填信息直接点提交会返回 500 错误
> ### 高优先级问题（1 个）
> 2. 支付成功后跳转到 404 页面，应该跳转到订单详情页
> ### 中优先级问题（1 个）
> 3. 移动端支付按钮只有 32px，达不到 44px 的最小触摸目标要求，容易误触
>
> 初始健康评分：65 分
>
> 我已经自动修复了这 3 个问题：
> 1. 给支付表单添加了非空校验，错误提示友好
> 2. 修复了支付成功后的跳转路径，改为跳转到对应订单的详情页
> 3. 把移动端按钮高度调整为 48px，增加点击热区
> 每个修复都单独提交了 commit，自动生成了对应的回归测试用例。
>
> 重新测试结果：所有问题都已修复，健康分提升到 92 分，没有发现新的问题。
>
> 测试报告已保存，包含所有问题的前后对比截图和复现步骤，现在可以合并代码上线。

## 七、迁移价值

这个角色的设计思路完全可以脱离 Gstack 复用：

- "测试→修复→验证"一站式的流程可以复用在团队的测试流程里，减少测试和开发之间的沟通成本，大大缩短 bug 修复周期
- diff 感知测试的思路可以用在 CI 流程里，只测试代码改动影响到的功能，大大节省 CI 测试时间
- 8 大类加权健康评分体系可以直接作为团队的产品质量标准，量化产品质量
- 核心逻辑是「测试不仅要发现问题，还要直接低成本解决问题，缩短反馈周期」，适用于所有需要做功能测试的产品研发场景
