## 一、这是一个什么工具

**一句话答案：**
Gstack 是一套 AI 编程工作流系统，把 AI 编程工具（Claude Code）变成一个虚拟工程团队。

不是一堆 prompt 集合，而是一套**有结构、有流程、有分工**的操作系统。

---

## 二、Garry Tan 为什么做这个

他引用了 Andrej Karpathy 的话：

> "I don't think I've typed like a line of code probably since December." — Andrej Karpathy，2026年3月

Karpathy 从12月起基本不写代码了。这个现象触动了 Garry。

Garry 自己在 60 天内输出了 **60 万行生产代码**，同时全职运营 YC。注意两个数字：

| 数字 | 含义 |
|------|------|
| 35% | 其中 35% 是测试代码 |
| 600,000+ 行 | 60 天，约 10,000-20,000 行/天 |

他不是全职在写代码——是**用对了工具**。

## 三、gstack 的核心理念

### 3.1 工程的壁垒已经消失

Garry 在 README 里画了一张表：

| 任务类型 | 传统开发 | AI 辅助开发 | 压缩比 |
|---------|---------|------------|--------|
| 脚手架/模板 | 2天 | 15分钟 | ~100x |
| 测试编写 | 1天 | 15分钟 | ~50x |
| 功能实现 | 1周 | 30分钟 | ~30x |
| Bug修复+回归测试 | 4小时 | 15分钟 | ~20x |
| 架构/设计 | 2天 | 4小时 | ~5x |
| 研究/探索 | 1天 | 3小时 | ~3x |

**结论：工程已经是最简单的部分。**

这意味着什么？
- 以前"够用就行"的90分方案，是因为完整实现太贵
- 现在完整实现只贵几秒钟，应该永远选完整方案

### 3.2 Boil the Lake（把湖烧开）

这是 Garry 的第一个核心方法论。

**Lake（湖）vs Ocean（海洋）：**
- **Lake**：可以烧开的——100%测试覆盖、所有边界情况、完整错误路径
- **Ocean**：烧不开的——重写整个系统、多季度迁移

**行动含义：** 选方案时，永远选完整实现的那个。如果 A 比 B 只多 70 行代码，**选 A**，因为完整实现的边际成本已经接近零了。

**反模式：**
- ❌ "B 方案 80% 够了，少写代码" → 如果 A 只多 70 行，选 A
- ❌ "测试以后再加" → 测试是成本最低的"湖"，应该最先烧开
- ❌ "这要两周" → 换成："人 2 周 / AI 辅助 1 小时"

### 3.3 Search Before Building（建前先搜）

第二个核心方法论。

**三层知识：**

| 层级 | 含义 | 风险 |
|------|------|------|
| Layer 1 | 业界标准模式，刻在 DNA 里的知识 | 假设"常识"一定对 |
| Layer 2 | 博客、流行实践、趋势 | 人类容易狂热，搜索结果被操纵 |
| Layer 3 | 第一性原理，从问题本身推导 | 最有价值，但最难 |

**最有价值的发现：**

不是找到一个方案去复制，而是：
1. 理解别人在做什么以及**为什么**
2. 用第一性原理质疑他们的假设
3. 发现传统方法**错在哪里**

这是"11分"时刻——zig while others zag。

### 3.4 Build for Yourself

第三个核心方法论。

**最好的工具解决自己的真实问题。**

gstack 不是 Garry 为了"做一个 AI 编程框架"而做的，是因为他想解决自己真实的工作效率问题。每个功能都是因为**需要了才做的**。

真实问题的具体性 > 假设需求的通用性。

## 四、28个技能体系详解

### 4.1 工作流循环：Think → Plan → Build → Review → Test → Ship → Reflect

这是 gstack 的**核心骨架**。每个技能各司其职，按顺序接力。

```
Think（思考）
    ↓
Plan（规划）
    ↓
Build（构建）
    ↓
Review（审查）
    ↓
Test（测试）
    ↓
Ship（发布）
    ↓
Reflect（复盘）
```

**关键：每个步骤的输出，是下一个步骤的输入。**

`/office-hours` 输出的设计文档，自动流向 `/plan-ceo-review`。`/plan-eng-review` 输出的架构图和测试计划，被 `/qa` 承接。没有东西会掉进缝隙里。

### 4.2 流程技能详解

#### Think（思考阶段）

**`/office-hours` — YC 导师**

不是帮你写代码，是**帮你重新定义问题**。

六个强制问题：
1. 你的用户真正的痛点是什么？
2. 这个痛点是真实存在的还是想象的？
3. 现状是什么，人们是怎么凑合的？
4. 你的方案为什么比别人好？
5. 如果这个方案失败了，最可能的原因是什么？
6. 最小可行的第一步是什么？

Garry 的原话：*"你说的是'日报应用'，但你描述的其实是'首席运营官 AI'*。这就是 `/office-hours` 在做的事——挑战你的假设，而不是执行你的指令。

#### Plan（规划阶段）

**`/plan-ceo-review` — CEO/创始人**

重新思考问题，找到隐藏在需求里的 10 星产品。四种模式：
- Expansion（扩展）
- Selective Expansion（选择性扩展）
- Hold Scope（保持范围）
- Reduction（缩减）

**`/plan-eng-review` — 工程经理**

锁定架构。输出：
- ASCII 数据流图
- 状态机
- 错误路径
- 测试矩阵
- 失败模式分析

**`/plan-design-review` — 资深设计师**

给每个设计维度打分（0-10），解释 10 分是什么样子的，然后修改方案让它达到 10 分。**AI Slop 检测**——识别"看起来不错但没有灵魂"的设计。

**`/design-consultation` — 设计合伙人**

从零构建设计系统。研究竞品，提出安全选择和创意风险，生成逼真的产品原型，写出 `DESIGN.md`。

#### Build（构建阶段）

**`/review` — 高级工程师**

找那些**通过 CI 但会在生产环境爆炸的 bug**。不是语法错误，是：
- 并发问题
- 边界条件
- 错误处理不完整
- 性能退化

自动修复明显的 bug，需要人工判断的才上报。

**`/investigate` — 调试工程师**

系统性根因调试。铁律：**不调查就不修**。追踪数据流，测试假设，3次修复失败后停止。

#### Test（测试阶段）

**`/qa` — QA 负责人**

这是 Garry 说的**最大的突破**。

用真实浏览器打开页面，发现 bug，直接修复，生成回归测试，重新验证。**AI 有眼睛了。**

关键：每个 bug 修复都会生成对应的回归测试，确保下次不会重现。

**`/qa-only` — QA 报告员**

同上，但只报告不修改代码。适合想让人工 review bug 清单的场景。

#### Ship（发布阶段）

**`/ship` — 发布工程师**

同步 main → 运行测试 → 审计覆盖率 → 推送 → 开 PR。

如果没有测试框架，`/ship` 会**自动帮你搭起来**。这是"Boil the Lake"的体现——测试覆盖率应该从第一天就有。

**`/land-and-deploy` — 发布工程师**

从"批准"到"生产验证"一步完成。合并 PR → 等 CI → 验证生产健康。

**`/canary` — SRE**

发布后持续监控。监控：
- 控制台错误
- 性能退化
- 页面加载失败

**`/benchmark` — 性能工程师**

建立页面加载、Core Web Vitals、资源大小的基准线。每次 PR 对比前后性能。

#### Reflect（复盘阶段）

**`/retro` — 工程经理**

每周复盘。数据驱动：
- 每个人的产出分解
- 提交流量（shipping streak）
- 测试健康趋势
- 成长机会

`/retro global` 可以跨所有项目、所有 AI 工具运行。

### 4.3 安全工具

| 技能 | 作用 |
|------|------|
| `/careful` | 危险命令前警告（rm -rf, DROP TABLE, force-push） |
| `/freeze` | 锁定编辑范围，防止改到无关代码 |
| `/guard` | `/careful` + `/freeze` 一次激活 |
| `/unfreeze` | 解除锁定 |

### 4.4 浏览器工具

这是 gstack 最独特的部分之一。

**`/browse` — QA 工程师**

给 AI 装上眼睛。真实 Chromium 浏览器，真实点击，真实截图，约 100ms/命令。

**`$B connect`** 启动带 UI 的真实 Chrome，你可以看着 AI 操作用户的屏幕。

**`$B handoff`** 当 AI 遇到 CAPTCHA 或 MFA 时，人类接管同一 Chrome 解决，AI 继续。

### 4.5 并行 Sprint

这是 gstack 最有想象力的部分。

**一个人管理 10-15 个并行 Sprint。**

每个 Sprint 跑在独立的 Claude Code 会话里，有自己的 workspace。通过 [Conductor](https://conductor.build) 管理并行。

- 一个会话在做 `/office-hours` 定义新想法
- 一个在做 `/review` 审查 PR
- 一个在实现功能
- 一个在做 `/qa` 测试 staging
- 六个在其他分支上

**没有流程，10 个 Agent 是 10 个混乱源。有流程，每个 Agent 知道该做什么、什么时候停。**

## 六、gstack 不是什么

**不是 prompt 集合。**
每个技能不是"帮我写代码"这种通用 prompt，而是针对特定角色、特定阶段、特定目标的结构化流程。

**不是 copilot。**
Copilot 是副驾驶，给你建议让你拍板。gstack 是整个工程团队，你是 CEO 做决策。

**不是银弹。**
没有流程和判断力，gstack 只是加速制造垃圾代码的速度。

## 七、一句话总结

**一个人，用对了工具，可以比一个传统团队移动得更快。**

关键是：
- **Boil the Lake** — 做完整
- **Search Before Building** — 先研究后动手
- **28 个专职角色** — 各司其职不扯皮
- **并行 Sprint** — 一个人当一个团队用
