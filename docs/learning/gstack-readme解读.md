# Gstack README 解读

> 基于 Garry Tan 的 Gstack README 原文，2026-03 整理

---

## 一句话定义

**Gstack 是一个 AI 编程工作流系统**——通过 28 个专职角色，把 Claude Code（或其他 AI 编程工具）变成一个虚拟工程团队。

---

## Gstack 解决什么问题

YC CEO Garry Tan 的原话：

> "我用 Gstack 60 天输出了 60 万行生产代码，同时全职运营 YC。"

但这背后有一个更深的洞察：

**工程是最简单的部分。** AI 让代码生成的成本趋近于零，但真正的瓶颈变成了：
- **Taste（品味）** — 什么值得做
- **Judgment（判断力）** — 什么方案是对的
- **Willingness to do the complete thing（完整完成的意愿）** — 不走捷径

---

## 核心方法论

### 1. Boil the Lake（把湖烧开）

**核心观点：** AI 让完整实现的边际成本接近于零。

| 任务类型 | 传统开发 | AI 辅助开发 | 压缩比 |
|---------|---------|------------|--------|
| 脚手架/模板 | 2天 | 15分钟 | ~100x |
| 测试编写 | 1天 | 15分钟 | ~50x |
| 功能实现 | 1周 | 30分钟 | ~30x |
| Bug修复+回归测试 | 4小时 | 15分钟 | ~20x |
| 架构/设计 | 2天 | 4小时 | ~5x |
| 研究/探索 | 1天 | 3小时 | ~3x |

**行动含义：** 以前选"够用就行"的90分方案是因为完整实现太贵。现在完整实现只贵几秒钟，应该永远选完整方案。

**前提区分：**
- **Lake（湖）** — 可以烧开的（100%测试覆盖、完整功能、所有边界情况）
- **Ocean（海洋）** — 烧不开的（重写整个系统、多季度迁移）

结论：烧湖，标注海洋。

### 2. Search Before Building（建前先搜）

**核心观点：** 第一个想法不是最好的，在动手之前先研究。

三层知识：
- **Layer 1** — 业界标准模式，深深刻在 distribution 中的知识
- **Layer 2** — 博客、最佳实践，要批判地看
- **Layer 3** — 第一性原理推理，最有价值的洞察

最有价值的发现不是找到方案，而是：
1. 理解别人在做什么以及为什么
2. 用第一性原理质疑他们的假设
3. 发现传统方案错误的清晰原因

### 3. Build for Yourself

**核心观点：** 最好的工具解决自己的真实问题。

真实问题的具体性 > 假设需求的通用性。

## 28 个技能体系

Gstack 把工作流分成 7 个阶段，每个阶段有多个专职角色：

### Think（思考）
- `/office-hours` — YC 导师，重新定义产品想法，6个强制问题

### Plan（规划）
- `/plan-ceo-review` — CEO，找到10星产品
- `/plan-eng-review` — 工程经理，锁定架构
- `/plan-design-review` — 设计师，评分设计维度
- `/design-consultation` — 设计合伙人，从零构建设计系统

### Build（构建）
- `/review` — 高级工程师，找通过CI但会在生产爆炸的bug
- `/investigate` — 调试工程师，系统性根本原因调试
- `/design-review` — 会写代码的设计师，审查+修复

### Test（测试）
- `/qa` — QA负责人，真实浏览器测试+修复+回归测试
- `/qa-only` — QA报告员，只报告不修改

### Ship（发布）
- `/ship` — 发布工程师，同步main→测试→推送→PR
- `/land-and-deploy` — 发布工程师，合并→等CI→验证生产
- `/canary` — SRE，发布后监控

### Monitor（监控）
- `/benchmark` — 性能工程师，Core Web Vitals对比

### Reflect（复盘）
- `/retro` — 工程经理，数据驱动的每周复盘

### 工具类
- `/document-release` — 技术写手，更新所有相关文档
- `/codex` — OpenAI Codex，双AI交叉审核
- `/autoplan` — 一键跑完CEO→设计→工程审核

## 安全工具

- `/careful` — 危险命令前警告
- `/freeze` — 锁定编辑范围
- `/guard` — 两者同时激活
- `/unfreeze` — 解除锁定

---

## 浏览器工具

- `/browse` — 真实Chromium浏览器，~100ms/命令
- `/setup-browser-cookies` — 从真实浏览器导入cookie
- `$B connect` — 启动带UI的真实Chrome，人类和AI同步控制

---

## 给谁用

- **创始人和CEO** — 特别是还想亲自ship的技术创始人
- **首次Claude Code用户** — 结构化的角色代替空白prompt
- **技术Lead和资深工程师** — 每个PR的严格review、QA和发布自动化

---

## 关键数字

- **60天** — 60万行生产代码
- **35%** — 其中35%是测试代码
- **10000-20000行/天** — 部分时间工作的产出
- **140,751行/周** — 3个项目合计
- **362 commits/周**

---

## 一句话总结

Gstack 证明了：**一个人用对了工具，可以比一个传统团队移动得更快。**

关键不是工具本身，而是背后的工作流程和思维方式。
