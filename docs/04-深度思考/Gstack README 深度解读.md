# Gstack README 深度解读

## 一、Gstack 是什么？
Gstack 是 Y Combinator CEO Garry Tan 开源的 **AI 原生软件工程流水线**，它把 Claude Code/Codex 等 AI 编程助手变成了一整套虚拟工程团队，实现了从产品设计、架构评审、代码开发、测试验证到上线发布的全流程自动化，让单个开发者具备媲美 20 人团队的研发效率。

Garry Tan 自己就是这套体系的最佳实践者：
- 全职管理 YC 的同时，60 天写出 60 万行生产代码（35% 是测试），日均输出 1-2 万行代码
- 单周净增代码量可达 11.5 万行，产出是 2013 年他 solo 开发 YC 内部社交网络 Bookface 时的 3 倍以上
- 目前已经可以同时支撑 10-15 个并行项目的开发

## 二、核心设计理念：把研发流程标准化成可执行的 AI 指令
Gstack 不是零散的工具集合，而是一套完整的研发流程规范，它把传统的软件工程 sprint 流程拆分成了 7 个标准化阶段，每个阶段对应专门的 AI 技能：
```
Think（思考）→ Plan（规划）→ Build（开发）→ Review（评审）→ Test（测试）→ Ship（发布）→ Reflect（复盘）
```
每个阶段的输出会自动作为下一个阶段的输入，完全避免了传统 AI 编程中「上下文丢失、流程断层、结果不可控」的问题。比如 `/office-hours` 生成的产品设计文档会自动被 `/plan-ceo-review` 读取，`/plan-eng-review` 生成的测试计划会自动被 `/qa` 使用，整个流程没有信息断层。

## 三、技能体系：28 个虚拟角色，覆盖研发全流程
Gstack 总共包含 **28 个技能**，分为「业务角色技能」和「效率工具技能」两大类：

### 1. 业务角色技能（21 个）
对应传统研发团队中的不同岗位角色：
| 技能 | 虚拟角色 | 核心职责 |
|------|----------|----------|
| `/office-hours` | YC 创业导师 | 产品需求梳理，挑战不合理的需求假设，输出产品设计文档 |
| `/plan-ceo-review` | 产品 CEO | 从用户价值角度评审产品方案，做 scope 控制和优先级排序 |
| `/plan-eng-review` | 技术经理 | 架构设计评审，输出数据流图、状态机、测试矩阵、风险评估 |
| `/plan-design-review` | 资深设计师 | UI/UX 设计评审，AI 垃圾设计检测，输出设计改进方案 |
| `/design-consultation` | 设计合伙人 | 从零搭建完整设计系统，生成产品 mockup，做竞品分析 |
| `/review` | 资深工程师 | 代码评审，自动修复低级 bug，识别代码完整性问题 |
| `/investigate` | 调试专家 | 系统化根因分析，禁止无排查直接改代码 |
| `/design-review` | 全栈设计师 | 设计评审+代码修复，输出前后对比截图 |
| `/design-shotgun` | 设计探索者 | 生成多套设计方案，打开对比页面供用户选择 |
| `/qa` | 测试负责人 | 自动化测试，自动修复 bug，生成回归测试用例 |
| `/qa-only` | 测试专员 | 只输出 bug 报告，不修改代码 |
| `/cso` | 首席安全官 | OWASP Top 10 + STRIDE 威胁建模，输出可利用的漏洞场景 |
| `/ship` | 发布工程师 | 代码合并、测试运行、覆盖率检查、提交 PR，自动生成测试框架 |
| `/land-and-deploy` | 运维工程师 | PR 合并、CI 等待、生产部署、健康检查，一键上线 |
| `/canary` | SRE | 发布后监控，检测控制台错误、性能回归、页面失败 |
| `/benchmark` | 性能工程师 | 页面性能基准测试，Core Web Vitals 对比 |
| `/document-release` | 技术文档工程师 | 自动更新所有项目文档，修正过时的 README 等内容 |
| `/retro` | 技术经理 | 项目复盘，产出个人产出统计、测试健康趋势、成长建议 |
| `/browse` | 测试工程师 | 真实 Chromium 浏览器操作，点击、截图、数据提取 |
| `/setup-browser-cookies` | 会话管理员 | 导入本地浏览器 Cookie，测试需要登录的页面 |
| `/autoplan` | 评审流水线 | 自动执行 CEO→设计→技术评审，只把需要人工决策的内容抛出来 |

### 2. 效率工具技能（7 个）
辅助提升研发效率和安全性的工具：
| 技能 | 核心功能 |
|------|----------|
| `/codex` | 跨 AI 模型二次评审，用 OpenAI Codex 做独立代码审查 |
| `/careful` | 安全护栏，执行危险命令（rm -rf、DROP TABLE 等）前预警 |
| `/freeze` | 编辑锁，限制 AI 只能修改指定目录的文件 |
| `/guard` | 组合安全模式，`/careful` + `/freeze` 二合一 |
| `/unfreeze` | 解除编辑锁 |
| `/connect-chrome` | 连接本地 Chrome 浏览器，实时观看 AI 操作过程 |
| `/setup-deploy` | 部署配置，自动识别平台和部署命令 |
| `/gstack-upgrade` | 自动升级 Gstack 到最新版本 |

## 四、核心特性解析
### 1. 多并行 Sprint 能力
Gstack 支持同时运行 10-15 个并行项目，每个项目跑独立的 sprint 流程，AI 会自动判断每个项目需要的评审类型，不需要人工干预，就像一个 CEO 同时管理多个团队一样，只需要关注最重要的决策点即可。

### 2. 真实浏览器自动化
`/browse` 技能用真实的 Chromium 浏览器操作，支持：
- 100 ms 级别的操作响应
- 连接本地 Chrome 窗口，实时观看 AI 操作
- 导入本地 Cookie，测试登录态页面
- 遇到验证码、MFA 等 AI 无法处理的场景时，自动把控制权交还给用户，处理完再恢复执行

### 3. 多 AI 交叉评审
`/codex` 技能支持调用 OpenAI Codex 做独立的代码评审，和 Claude 的评审结果做交叉对比，找出两个模型各自发现的问题，大幅降低 bug 漏检率。

### 4. 全链路测试保障
- `/ship` 会自动为没有测试框架的项目生成测试框架
- 每次发布都会做覆盖率审计
- 每个 `/qa` 发现的 bug 都会自动生成回归测试用例
- 目标是 100% 测试覆盖率，让 AI 写代码不再是"裸奔"

### 5. 自动文档更新
`/document-release` 会自动扫描项目所有文档，对比代码 diff，自动更新过时的内容，README、架构文档、贡献指南都会自动保持最新，不需要人工维护。

## 五、适用人群
Gstack 并不是给纯新手准备的工具，它最适合三类人群：
1. **技术创始人/CEO**：懂技术的创始人可以用它快速验证产品想法，不用依赖技术团队，自己就能快速迭代产品
2. **技术负责人/资深工程师**：可以用它做架构评审、代码审查、自动化测试，大幅提升团队研发效率
3. **独立开发者/自由职业者**：一个人就能干前端、后端、测试、运维的活，把精力集中在最有价值的产品设计上

## 六、总结
Gstack 代表了 AI 时代软件工程的新方向：把人的经验沉淀成标准化的流程，让 AI 去执行流程中的重复性工作，人只需要做最重要的决策。就像工业革命让机器代替了手工劳动，Gstack 这样的工具会让 AI 代替大部分重复性的编程劳动，未来的开发者会更像产品经理和架构师，而不是代码的搬运工。
